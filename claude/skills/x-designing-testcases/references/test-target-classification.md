# テスト対象の特性判別

テストコードを書く前に、対象がどの特性カテゴリに属するかを判別します。判別結果に応じて次に読むpattern referenceが決まります。

## 判別フロー

```
対象を Read
   ↓
段階1: 振る舞いテストの対象か？
   ↓
段階2: どのインターフェイスを通すか？
   ↓
段階3: 純粋ロジックなら業務ルールを持つか？
   ↓
カテゴリ確定 → 対応する pattern reference を読む
```

## 段階1: 振る舞いテストの対象か

以下は振る舞いテストの対象外です。`designing-testcases`のスコープ外です。

| 対象外 | 担当 |
|---|---|
| 型エラー | `tsc` / IDE / コードレビュー |
| API スキーマ整合 | contract ツールの設定 |
| バンドルサイズ / ESM/CJS 互換 | ビルド設定 / size-limit |
| 静的a11y違反（color-contrast、button-name、aria属性の存在） | axe / markuplint 等のリンター |

これらの観点はテストではなく品質ゲートとして担保します。テストを書く過程で自然に検知されます。

## 段階2: インターフェイス判別

対象が通すインターフェイスで分類します。

### Reactに依存するか

JSX、`useState`、`useEffect`、`useContext`等への依存を確認します。

- **Yes（React必要）** → 段階2Aへ
- **No（純粋TypeScriptで書ける）** → 段階2Bへ

### ブラウザ全体・サーバー側の状態が関わるか

ページ遷移、複数コンポーネント間の協調、DB変更、外部API呼び出し、CDN・キャッシュなどが関わるか。

- **ユーザージャーニーとして観察する** → `e2e-test-patterns.md`
- **視覚的な差分を観察する** → `visual-test-patterns.md`
- **キーボード・フォーカスの振る舞いを観察する** → `a11y-test-patterns.md`
- **サーバー側の状態変化を観察する（DB / 監査ログ / 権限境界 / 並行操作）** → `integration-test-patterns.md`

### 段階2A: React依存コードの分類

- レンダリングが主な責務（propsを受けてUIを出す）→ **コンポーネント** → `component-test-patterns.md`
- 副作用・状態管理が主な責務（外部API呼び出し、localStorage、subscription等）→ **hooks** → `hooks-test-patterns.md`

### 段階2B: 純粋ロジックの分類

段階3へ進みます。

## 段階3: 純粋ロジックの内訳

「業務が定めたルール・制約・変換」を持つかを問います。

| 種別 | 意味 | 例 |
|---|---|---|
| **判定** | 業務のルールに基づきtrue/falseを返す | `canAccessTipster(isRaceFinished, permission)` |
| **制約** | 業務上あり得ない値を型・関数で弾く（Branded type） | `HorseNumber`は1〜99の整数のみ |
| **変換** | 業務の知識が必要な値の導出・変換 | 券種コード → 表示名、予想印ソート順 |

- **Yes（業務ルールあり）** → `domain/`配下 → `domain-test-patterns.md`
- **No（業務ルールなし、汎用処理）** → `utils/`配下 → `utils-test-patterns.md`

## 曖昧なケースの処理

段階2-3の判定で決まらないケースは、追加の観点を使います。

### 核心的な判断軸（最初に問う）

> **UIや特定の画面がなくなったとしても、そのルールは意味を持つか。**

「表示する/しない」はその画面が存在するから意味を持つ概念であり、業務ルールではなくUX設計が決めることです。コンポーネントまたは画面レベルのロジックとして扱います。個々の条件判定（権限の確認・状態の判定）はドメインロジックとして切り出せる場合がありますが、それらを「この画面で表示するかどうか」に組み合わせる判断はプレゼンテーションロジックです。

### `domain/` vs `utils/`の区別

「このプロジェクト以外でも使えるか？」を問います。

- **業務の知識が必要** → `domain/`（アプリ固有の概念）
- **どのプロジェクトでも使える純粋な汎用処理** → `utils/`（文字列操作・日付フォーマット・数値丸め等）
- **外部ライブラリのラッパー・設定・クライアントインスタンス** → `lib/`（axiosインスタンス・QueryClient設定等）。`lib/`は通常テスト対象外（ラッパーの薄さで担保）

### FEで持つべきかの確認

BEのビジネスルールをフロントが独自に再実装していないかを確認します。

| 判断軸 | FEで持つ | BEに任せる |
|---|---|---|
| UXへの影響 | リアルタイムフィードバックが必要（フォームバリデーション等）→ FEに持つ（BEにも必ず持つ） | バッチ処理・非同期計算 |
| 一貫性の必要性 | UI表示専用のフォーマット（日付・通貨表示）→ utils/に置く | 複数クライアント間で一致が必要なルール |
| セキュリティ | 補助的な検証（UX向上のみ） | 認可・セキュリティ上の検証は必ずBEにも持つ |
| 複雑性 | 単純なルール（条件が少ない） | 複雑なビジネスルール（多条件・データ依存） |

判断フロー:
- リアルタイムフィードバックが必要 → FEにバリデーションを持たせます（BEにも必ず持たせます）。不要ならBEに任せます
- UI表示専用のフォーマット変換 → `utils/`に置きます（ドメインロジックではなくユーティリティ）
- 複数クライアントで一致が必要 → BEで管理し、FEはAPIの結果を使います

### 権限・アクセス制御

`canEdit(role)`のように「APIのないシステムでも成立するルール」はドメインロジックです。フロントが権限ロジックを独自に再計算している場合は`domain/`に置きません。

- OK: `canAccessTipster(isRaceFinished, userPermission)`— BEの`isDenied`を参照するだけ
- NG: プランの種類ごとの機能権限をフロントが独自に計算する

### URLパス・メニュー名称

原則は`constants/`（テスト対象外）です。ただしキー型に業務上の意味があってナビゲーション構造が業務的な分類を表している場合は`domain/`に置いても構いません。

### 表示用定数（色・アイコン）と業務ルールが混在する場合

`displayPriority`（業務的な順序ルール）と`color`（表示用）が同じオブジェクトにあって分離するとかえって複雑になる場合は、まとめて`domain/`に置いても構いません。

## 配置スコープの判断（参考）

`domain/`に置くことが決まった場合の配置スコープです。これは配置ルールであってテスト対象判別とは別の話ですが、配置先を確認するときに参照することがあります。

| 状況 | 配置先 |
|---|---|
| そのfeatureだけで使う | `src/features/{feature}/domain/` |
| 複数の独立したfeatureにまたがる | `src/features/common/domain/` |
| 同一featureグループ内の複数サブfeatureにまたがる | `src/features/{group}/common/domain/` |
| 設計当初から共通利用を意図している | 1箇所でしか使わなくても`common/domain/`に置いてよい |

## 落とし穴

| 落とし穴 | 対処 |
|---|---|
| ファイル名や置き場所だけで判別する | `utils/`に置いてあっても業務ルールを持つことがあります（過去の整理漏れ）。内容を読んで判別します。 |
| 「両方の特性を持つ」と判定する | 関数が複数の責務を持っている可能性が高いです。設計改善の対象として提案します。 |
| 静的a11y違反をテストに書く | リンター責務です。`a11y-test-patterns.md`には動的な振る舞いだけ書きます。 |
| 型で防げるものをテスト対象にする | 段階1のフィルターで除外します。型に寄せて実装する方針を提案します。 |
