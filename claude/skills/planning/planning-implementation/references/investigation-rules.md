# Investigation Rules（調査ルール）

`planning-implementation` スキルのPhase 2「コードベース調査」で使う詳細ルールです。

調査範囲を制御する目的は二つあります。

- コンテキストの浪費を防ぐ: 関係ないファイルを読み込んでコンテキスト窓を埋めません
- 調査不足を防ぐ: 必要な前提を読まずに計画を書き始めません

両方を同時に達成するため、Tier別に「読むべき度合い」を分け、完了判定を明確にします。

## 読むべき対象（Required Reading）

### Tier 1: ほぼ常に読むもの

タスクの種類に関わらず、以下はまず読みます。これを飛ばすと既存パターンに反した計画を書きがちです。

- `package.json` — 依存ライブラリとscripts
- `tsconfig.json`（TypeScriptプロジェクトの場合）— パスエイリアス、コンパイラオプション
- ルートの `README.md` — プロジェクト全体の意図と起動方法
- 該当featureのディレクトリ構造（`ls` または `tree -L 2` 程度）
- 該当featureに `README.md` があれば読みます
- ESLint/Biome/Prettierの設定 — コードスタイル制約

### Tier 2: 該当タスクで関係するなら読むもの

- 既存のADR（Architecture Decision Records）— 過去の設計判断の根拠
- 類似実装が存在する場合のそのファイル — 倣うべきパターンの抽出
- 関連するhooks/utils/共通コンポーネント — 再利用候補の特定
- 同種機能のテストファイル — テストの書き方規約
- `Constraints` に書く可能性のあるドキュメント（命名規約、コミット規約など）

### Tier 3: 必要に応じて辿るもの

- `git log` での直近のコミット履歴 — 該当領域がいま動いているか
- 関連PR — レビュー観点や直近の議論
- 隣接featureの実装 — 横断的な制約があるかの確認

Tier 3に深入りする前に、Tier 1〜2の情報で計画が書けないかを必ず確認してください。

## 読まない対象（Skip List）

以下は明示的に読みません。トークンの無駄になるためです。

- `node_modules` 配下
- ビルド出力（`dist`、`.next`、`build`、`out`、`.turbo` など）
- lockfile（`package-lock.json`、`pnpm-lock.yaml`、`yarn.lock`）
- 関連が薄いfeatureのディレクトリ
- 古いマイグレーション履歴（直近以外）
- 自動生成ファイル（`*.generated.*`、GraphQL Codegen出力など）

## 読み順の戦略

### Step 1: Breadth-first（全体俯瞰）

最初に `ls` や `tree` で構造を把握します。「どのファイルが関係しそうか」のあたりをつけます。

### Step 2: Targeted-deep（該当ファイルの精読）

あたりをつけたファイルを読みます。タスクと無関係な行は読み込みません。

### Step 3: Reference-depth（参照の追跡）

精読中に出てきた `import` を必要に応じて辿ります。ただし3階層以上の追跡は禁止です。3階層辿っても答えが出ないなら、調査範囲の取り方が間違っている可能性が高いため、Step 1に戻って候補を再検討してください。

## 一度に読むファイル数の上限

1セッションで20ファイル以下を目標とします。超える場合は、それ以上読み進める前に「どのファイルから何の情報を得たか」を整理してください。整理せずに読み続けるとコンテキストが飽和して判断力が落ちます。

## 完了判定

以下をすべて言語化できれば調査完了です。1つでも言語化できないなら調査不足です。

- このタスクで触る予定のファイル候補（少なくとも1〜3個）
- 既存の類似実装と、そのうち倣うべき点と倣わない点
- 既存の制約（依存方向、命名規約、テスト規約、スタイル規約）
- 触るべきでない領域（変更禁止のファイルやモジュール）
- このタスクで生まれる新規ファイルの配置先候補

言語化を求める意図は「読んだつもりで読めていない」を検出するためです。脳内で「なんとなくわかった」では合格にしません。

## アンチパターン

| アンチパターン | 内容 |
|---|---|
| grepで網を広げすぎる | 関連語をgrepした結果を全件読み込んでしまうパターンです。最も関連性が高そうな1〜3件に絞ります |
| importの再帰的全追跡 | `A.ts` の `import` を全部辿り、その先の `import` をさらに辿るパターンです。Step 3のルール（3階層まで）を守ります |
| `node_modules` に入る | 型定義の確認は `.d.ts` の該当部分のみで止めます |
| テストファイルの全読 | 同種機能のテストファイルだけ読みます。それ以外は読みません |
| Tier 3を先に読む | `git log` やPRから入ると文脈に飲まれます。Tier 1から順に読みます |

## Phase 7レビューでの追加調査

Phase 7のレビューで「調査が浅い」「前提の確認漏れ」と判定された場合、このルールに戻って追加調査を行います。

見落としが起きやすいパターンは以下の通りです。

- Tier 1の見落とし（特にtsconfigのパスエイリアスやESLint設定）
- 類似実装の調査漏れ（既に同じことをやっているコードがある）
- 変更禁止領域への影響の見落とし

これらが該当する場合は、対応するTierに戻ってやり直してください。
