# Review Criteria（レビュー観点）

`planning-implementation` のPhase 7で使う、実装計画固有のレビュー観点リストです。AIコーディングエージェント向け実装計画に特化した観点を整理しています。

## このドキュメントの位置づけ

実装計画のレビューには次の特有の前提があります。

| 前提 | 内容 |
|---|---|
| 読み手はAIコーディングエージェント | 曖昧さを「もっともらしい解釈」で埋め、コンテキストにない情報を「あるはず」と仮定し、抽象的な完了条件を独自に「達成した」と判断する可能性があります |
| 目的は実行 | ここでは「実装に着手できるか」を見ます。未解決の論点があれば、その箇所を指摘します |
| 構造は固定 | `planning-implementation` のテンプレート（Context / Goal / Non-goals / Constraints / References / Acceptance Criteria / Decisions Log / Open Questions for Implementation）にしたがっています |

これを踏まえて、以下の観点で検証します。

## レビュー観点

### 1. Context（前提・コードコンテキスト）の正確性

- 列挙されたファイルパス・ディレクトリは実在するか
- 技術スタックのバージョンは実際の `package.json` 等と一致するか
- 関連featureとして挙げられたものに、本当に関連性があるか
- 関連の薄いfeatureが含まれていないか（コンテキスト浪費の温床）
- 関連ドキュメント・ADRのリンクは生きているか

確認方法は `ls`、`find`、`cat package.json` 等で実在性を検証してください。

### 2. Goal（目的・成果物）の観測可能性

- 成果物単位で書かれているか。動機・形容詞だけになっていないか
- 「達成された状態」が観測可能な形で書かれているか
- Acceptance Criteriaと対応関係が成立するか（Goalが達成されればCriteriaが満たされる関係）

### 3. Non-goals（含めないこと）の具体性

- 具体的なファイル/機能名で書かれているか
- 「不要な変更をしない」「過剰な最適化はしない」のような抽象表現が残っていないか
- AIエージェントが「ついでにリファクタリング」する誘惑に勝てる粒度か
- 隣接機能への波及防止が含まれているか

### 4. Constraints（制約）の機械検証可能性

- 既存ルール/ADRへの参照は具体的なパスか
- 依存方向のルールは機械的に判定できる形か
- 変更禁止ファイルが具体的なパスとして列挙されているか
- 性能・セキュリティ要件は具体的な閾値で書かれているか
- 「使用してよい/してはいけないライブラリ」のリストが片寄っていないか

### 5. References（参考実装）の実在性と妥当性

- 列挙されたファイルパスは実在するか
- 「倣うべき点」が具体的に書かれているか（漠然と「これを参考に」で終わっていないか）
- 古い・非推奨パターンを参照していないか
- 複数のReferences間で矛盾するパターンを示していないか

確認方法は参照ファイルを実際に開き、推奨パターンが現行のコードベースと整合するか検証してください。

### 6. Acceptance Criteria（完了条件）の検証可能性

観点9と並ぶ最重要観点です。

- 曖昧語（`適切` / `いい感じ` / `ちゃんと` / `きちんと` / `うまく` / `なるべく` / `可能な限り` / `必要に応じて`）が残っていないか
- 検証コマンドが `package.json` 等に存在するscriptを指しているか
- 各Criteriaに検証コマンドまたは検証手順が紐づいているか
- AIが「PASSしたつもり」になりやすい条件（型チェックだけ、コンパイル通過だけなど）になっていないか
- エッジケース・エラーハンドリング・境界値のCriteriaがあるか
- ユニットテストを書く実装の場合、各関数の具体的な入出力例が列挙されているか
 （例: `getServiceDate(2025/12/1 04:59) → 2025/11/30`）
- 境界値テストの「手前・ちょうど・超え」と「月跨ぎ・年跨ぎ」など境界を越える状態遷移が網羅されているか
- 検証単位が十分小さいか

確認方法は以下のコマンドで曖昧語を検出してください。

```bash
grep -nE "適切|いい感じ|ちゃんと|きちんと|うまく|なるべく|可能な限り|必要に応じて" <計画ファイル>
```

### 7. Decisions Log（判断記録）の根拠

- `Decided by code` の根拠としてファイルパス・行番号が書かれているか
- `Decided by default` の根拠が業界標準・既存パターンとして実在するか
- 「調査をサボった結果の適当な決定」が紛れ込んでいないか
- 判断と根拠が1対1で対応しているか（複数判断が1つの根拠でまとめられていないか）

### 8. Open Questions for Implementation（実装時の判断保留事項）の妥当性

- `if-then` 形式で具体的に書かれているか（「もし○○なら、△△する」）
- 計画段階で決められたはずの判断がここに先送りされていないか
- 実装中の即断としては大きすぎる事項が含まれていないか
- 数が多すぎないか（多い場合は計画の前提が薄いサインです）

### 9. AIエージェント特有の「思い込み」発生ポイント

最重要観点です。AIが計画を読んで実装するときの典型的失敗モードを検出します。

| 失敗モード | 発生条件 |
|---|---|
| 曖昧語の残留 | 「適切に」「いい感じに」が残ると、AIは独自解釈で実装します |
| 不在ファイルの捏造 | import先や参照ファイルが具体的に書かれていないと、AIは存在しないパスを捏造する可能性があります |
| テストの逆算実装 | テストの期待値や入出力例が書かれていないと、AIは「テストを通すための実装」を逆算で書きます |
| 過剰なエラーハンドリング | エラー処理方針が書かれていないと、AIは至るところに `try-catch` を撒く可能性があります |
| 独自命名による整合性崩壊 | 命名規約への参照がないと、AIは独自命名で進めてプロジェクトの一貫性を崩します |
| 暗黙のスコープ拡張 | Non-goalsが甘いと、AIは「関連しそうな箇所」まで手を伸ばします |

上記6項目の観点で計画を読み直し、AIが解釈の余地を持つ箇所を質問として返してください。

### 10. 計画のサイズ妥当性

- 1セッション（1エージェント・1PR）で完了できるサイズか
- Acceptance Criteriaが7個を超えていないか
- 性質の異なるタスクが混在していないか（例: データ構造変更とUI実装が同居）
- Epic→Subtaskに分割すべきサイズではないか

### 11. セクション間の整合性

- GoalとAcceptance Criteriaの対応関係が成立するか
- ConstraintsとReferencesの間で矛盾するパターンを示していないか
- Non-goalsとGoalが矛盾していないか（例: Goalで触る対象をNon-goalsで除外している）
- Decisions LogとConstraintsが矛盾していないか

## `planning-implementation` Quality Gatesとの対応関係

| Quality Gate | 主に関連する観点 |
|---|---|
| G1: 構造の完全性 | 1, 2, 3, 4, 5, 6, 7, 8 |
| G2: 具体性 | 1, 3, 4, 5 |
| G3: 検証可能性 | 6, 9 |
| G4: 整合性 | 11 |
| G5: サイズ | 10 |

Quality Gatesのセルフチェックでは見落としやすい観点として、とくに観点5（実在性）、9（思い込み）、11（整合性）に注力してください。

## 出力フォーマット

観点名は「1. Context...」「2. Goal...」など番号と日本語名で参照します。

強制度の基準は以下の通りです。

| 強制度 | 基準 | 例 |
|---|---|---|
| 必須 | 計画のままでは実装不能、またはAIが大きく誤解して実装する確率が高い | Acceptance Criteriaに曖昧語が残っている、参照ファイルが実在しない、Goalが観測可能でない |
| 推奨 | 実装は進められるが、品質や整合性に影響する | Non-goalsが抽象的、Decisions Logの根拠が薄い、Open Questionsが多すぎる |
| 任意 | 実装に直接影響しないが、再利用性や保守性で改善余地がある | References間の倣う点と倣わない点の整理、Decisions Logの記述スタイル |

## 質問の書き方

実装計画固有の補足は以下の通りです。

- AIが解釈の余地を持つ箇所は、「AIエージェントが○○と解釈する可能性があります」という形で具体化して指摘します
- 検証コマンドの実在確認は、「`<コマンド>` を実行して確認してみてください」と具体的に書きます
- 計画のサイズ過大を指摘する場合は、「分割案として○○と△△に分ける選択肢が考えられます」と分割例を添えます
