# writing-review

対象の `.md` ファイルを以下のルールに従ってレビュー・修正する。

## ルール

- h1からh6まで適切に使用する
- 引用やコードブロックを装飾として使わない
- コードブロック（```）は、ソースコード・コマンド・設定ファイル・実行結果など、そのまま機械が解釈する、またはそのまま端末に貼り付ける内容にのみ使う。手順・箇条書き・説明文をインデントや等幅フォントで整形する目的では使わない
- Markdownの引用記法（>）は他者の発言や外部資料からの引用にのみ使う。前提条件・注意書き・補足説明を目立たせる目的では使わない
- 前提条件や注意書きを区別したい場合は、見出し（##）またはGitHub Flavored Markdownのalert記法（> [!NOTE]等）を使う
- ドキュメントには信頼性のある情報のリンクと引用を積極的に記載する

  引用フォーマット:

  ```
  ＜英語の引用＞
  ＜日本語訳（日本語を引用する場合は省略）＞
  ＜[引用した情報のタイトル](URL)＞
  ```

- 1つの段落では1つの論点を扱い、論点が切り替わったら段落を分ける
- リスト項目の先頭に絵文字をラベルとして付けない（✅❌💡🔥🚀等）
- 述語で終わる文のあとにコロンを置いてブロック要素（リスト・コードブロック・引用・表）を続ける英語的なパターンは使わない。名詞で終わるラベル（「使用方法:」等）は許容する
- 冗長な表現は短くする（「することができます」→「できます」、「まず最初に」→「まず」、「する必要があります」→「してください」）
- 主語が不明な受動態は避け、動作の主体を明示する（「処理が行われます」→「システムが処理します」）
- 抽象的な形容に頼らず、定量的・具体的に書く（「高速な」→「50ms未満の」、「大幅に向上」→「従来比200%向上」）
- 同一の対象には同一の用語を使う（文書内で「ユーザー」と「クライアント」のように表記を揺らさない）
- 誇張表現・ハイプ的な表現は使わない（「革命的な」「ゲームチェンジャー」「究極の」「完全に解決」「パラダイムシフト」等）

## 引数

- 対象ファイルパス（必須）
- textlint 残存エラー（任意）: format.sh から自動呼び出しされた場合に渡される

## 手順

1. 対象ファイルを Read する
2. 上記「## ルール」の各項目をすべて確認する。形式的な変換が容易なルールだけでなく、意味的な判断が必要なルールも同等に確認する。段落分割は各段落の論点を読んで切り替わりを判断する。textlint 残存エラーが渡された場合は合わせて確認する
3. 検出した違反をすべて Edit で修正する
4. 修正内容の概要を報告する

## 注意

- 文体・構造・表現の修正のみを行う。技術的な内容（コード例・コマンド・仕様など）は変更しない
- ルール違反がなく textlint エラーもない場合は変更しない
