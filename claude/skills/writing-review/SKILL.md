# writing-review

対象の `.md` ファイルを以下のルールに従ってレビュー・修正する。

## ルール

### 差分チェック（Read不要）

差分の `+` 行のみを対象にパターンマッチで検出する。

- 水平線（---）を視覚的な区切りとして使わない。セクションの区切りは見出しで表現する
- 英数字と日本語の間に半角スペースを入れない（「Claude Code を使う」ではなく「Claude Codeを使う」）
- 太字（**）は使わない。強調が必要な箇所は文の構造や語順で伝える
- リスト項目の先頭に絵文字をラベルとして付けない（✅❌💡🔥🚀等）
- 冗長な表現は短くする（「することができます」→「できます」、「まず最初に」→「まず」、「する必要があります」→「してください」）
- 文は述語（動詞・形容詞・助動詞）で終える。名詞で文を終えて述語を省略しない。述語がないと時制（する／した）や意図（する／できる／したい）が曖昧になる
- 誇張表現・ハイプ的な表現は使わない（「革命的な」「ゲームチェンジャー」「究極の」「完全に解決」「パラダイムシフト」等）
- 調査や執筆の経緯（「以前調べたときは」「当初はXと考えましたが」「再調査した結果」等のパターン）は書かない。現時点で正しい結論だけを書く。仕様や事実が変わったことに意味がある場合は客観的な事実の変化として書く（「v2.0でAからBに変更された」等）

### 変更段落チェック（変更行を含む段落を Read）

変更行の前後にある空行を段落境界とし、その段落のみを Read して確認する。

- コードブロック（```）は、ソースコード・コマンド・設定ファイル・実行結果など、そのまま機械が解釈する、またはそのまま端末に貼り付ける内容にのみ使う。手順・箇条書き・説明文をインデントや等幅フォントで整形する目的では使わない
- Markdownの引用記法（>）は他者の発言や外部資料からの引用にのみ使う。前提条件・注意書き・補足説明を目立たせる目的では使わない。前提条件や注意書きを区別したい場合は、見出し（##）またはGitHub Flavored Markdownのalert記法（> [!NOTE]等）を使う
- コロン（:／：）で説明やブロック要素を導くパターンは使わない。ブロック要素を導く場合は直前の文を「以下の通りです。」等で終えて段落を分ける。項目と説明の対が短い場合は表を使い、各項目に段落が必要な場合は見出しと本文で構造化する
- 名詞を別の名詞に直接かける形（複合名詞）は使わず、助詞（の・による・に関する等）または活用形（した・する・な等）を補って修飾関係を明示する
- 主語が不明な受動態は避け、動作の主体を明示する（「処理が行われます」→「システムが処理します」）
- 抽象的な形容に頼らず、定量的・具体的に書く（「高速な」→「50ms未満の」、「大幅に向上」→「従来比200%向上」）
- 1つの段落では1つの論点を扱い、論点が切り替わったら段落を分ける

### 全体チェック（ファイル全体を Read）

format.sh経由の場合は、差分に見出し行（`#` で始まる行）または初出の用語が含まれる場合のみ実施する。直接呼び出しの場合は常に実施する。

- h1からh6まで適切に使用する
- 文末はです・ます調で統一する。だ・である調と混在させない
- 同一の対象には同一の用語を使う（文書内で「ユーザー」と「クライアント」のように表記を揺らさない）
- ドキュメントには信頼性のある情報のリンクと引用を積極的に記載する

  引用フォーマット:

  ```
  ＜英語の引用＞
  ＜日本語訳（日本語を引用する場合は省略）＞
  ＜[引用した情報のタイトル](URL)＞
  ```

## 対象の決定

呼び出し元によって対象の取得元が異なる。

### format.sh 経由（PostToolUse hook）

feedbackメッセージの「ファイル: `<path>`」を対象ファイルとする。

「変更差分:」セクションが含まれる場合は、diff の `@@` ヘッダーから変更行の範囲を特定し、その前後5行のみをレビュー対象とする。差分がない場合（新規ファイル等）はファイル全体を対象とする。textlint残存エラーが含まれる場合は合わせて確認する。

### 直接呼び出し

ユーザーの指示で明示されたファイルパスまたは範囲を対象とする。引数が `staged` の場合は `git diff --cached --name-only` で対象ファイルを取得し、`git diff --cached -- <path>` でステージング差分を取得する。

## 手順

1. 上記「## 対象の決定」に従い、対象ファイルパスとレビュー範囲を確認する
2. **差分チェック**: 差分（format.sh経由）またはユーザー指示の変更箇所の `+` 行に対して「差分チェック」ルールを確認する。Read不要
3. **変更段落チェック**: 変更行を含む段落を Read し、「変更段落チェック」ルールを確認する。textlint残存エラーがある場合は合わせて確認する
4. **全体チェック**: 「全体チェック」ルールの実施条件を判断し、必要な場合はファイル全体を Read して確認する
5. 検出した違反をすべて Edit で修正する
6. **ステージング乖離チェック**: ステージング対象ファイルを修正した場合、`git diff -- <path>` を実行する。ステージングにのみ存在する行（`-` 行）が生じている場合はユーザーに報告して修正を中断する
7. 修正内容の概要を報告する

## 注意

- 文体・構造・表現の修正のみを行う。技術的な内容（コード例・コマンド・仕様など）は変更しない
- ルール違反がなく textlint エラーもない場合は変更しない
- Edit は違反箇所のテキストのみを置換する。違反を含む段落・セクション・ファイル全体を new_string に置き換えるような広範な Edit は行わない
