# VERIFY コメント仕様

テストファイル内にAgentic Verificationと手動検証の情報を埋め込むためのコメント形式です。テストファイル自体を3層検証のSSoTにします。

## 設計意図

- 自動テスト = `it()` / `test()`の本体（CIで実行）
- `VERIFY:agentic` = エージェントがgrepして自律実行する検証
- `VERIFY:manual` = エージェントがgrepしてユーザーに渡す確認項目
- `VERIFY:meta` = テストファイル全体に効く文脈情報（UCファイルが残らないプロジェクトでもこれだけで機能仕様が復元できる）

3層が同じファイルに同居するため、開発中に「実は手動確認も要る」と気づいたら追記するだけで済み、別ファイルのメンテが不要になります。

## 共通ルール

- コメントは`// VERIFY:タイプ`で始めます（行頭の`//`の後にスペース1つ）
- 直後の行から`//   フィールド名:`の形式で続けます（`//`の後にスペース3つ + フィールド名）
- 1フィールド1行にします
- 空行または`//`で始まらない行が現れると1ブロック終了です
- フィールドの順序は仕様通りに揃えます（パースは順序に依存しませんが、可読性のため）
- 必須フィールドが欠落している場合、エージェントは警告を出します
- 任意フィールドはフィールドごと省略してよいです

## 配置場所

| 配置 | 効果範囲 |
|---|---|
| ファイル先頭 | ファイル全体 (`VERIFY:meta`を必ずここに置く) |
| `describe()`直上 | そのスイート全体 |
| `it()` / `test()`直上 | そのテストケース固有 |

## `VERIFY:meta` (ファイル先頭、SSoT)

```ts
// VERIFY:meta
//   対象機能:           プロフィール画像変更
//   想定ユーザー:       ログイン済みのエンドユーザー
//   目的:               プロフィール画像を新しい画像に置き換える
//   前提条件:           ログイン済み / 既存画像あり
//   起こり得る外部条件: cond.network (オフライン) / cond.external (ストレージAPI 5xx)
//   テスト範囲外:       内部APIの詳細 / モデレーション判定
//   最終レビュー日:     2026-05-13
```

| フィールド | 必須 | 内容 |
|---|---|---|
| `対象機能` | 必須 | 機能の説明（短く） |
| `想定ユーザー` | 必須 | 誰がこの機能を使うか |
| `目的` | 必須 | 達成したいこと |
| `前提条件` | 必須 | この機能が成立するための条件 |
| `起こり得る外部条件` | 必須 | ネットワーク・時刻・外部API等の前提（UC側の状況軸`cond.xxx`を含む） |
| `テスト範囲外` | 必須 | このファイルで扱わないこと |
| `最終レビュー日` | 必須 | `YYYY-MM-DD`形式 |

## `VERIFY:agentic`（describeまたはit直上）

```ts
// VERIFY:agentic
//   検証内容:               border-radiusがdesign tokenの値で適用されている
//   検証手順:               1. ブラウザMCPでプロフィール画面を開く 2. avatar要素をquerySelectorで取得 3. getComputedStyle().borderRadiusを取得
//   合格基準:               '9999px' または計算値が完全な円形になる値
//   自動テスト化しない理由: visual regressionは誤差吸収のため数値が見えない。実値を直接アサートしたい
//   自動テスト化する条件:   token経由ではなくハードコード値が再発したらe2eに昇格

test('プロフィール画像が円形で表示される', async ({ page }) => {
  // ...
});
```

| フィールド | 必須 | 内容 |
|---|---|---|
| `検証内容` | 必須 | 検証する内容 |
| `検証手順` | 必須 | 何をどの順番で実行するか |
| `合格基準` | 必須 | 合格判定基準 |
| `自動テスト化しない理由` | 必須 | なぜ自動テストではなくAgentic検証なのか |
| `自動テスト化する条件` | 任意 | どの条件を満たしたら自動テストに昇格させるか |

## `VERIFY:manual`（describeまたはit直上）

```ts
// VERIFY:manual
//   検証内容:           スクリーンリーダーでの読み上げ順序の自然さ
//   検証手順:           1. macOSのVoiceOverを有効化 (Cmd+F5) 2. プロフィール画面に遷移 3. VO+矢印キーで画面を読み上げる
//   合格基準:           「プロフィール画像」→「画像を変更ボタン」→「氏名入力」の順に読み上げられる
//   手動で確認する理由: axeでは読み順の意味的な自然さは検出できない
//   実施タイミング:     リリース前 / 該当画面のレイアウト変更時

test('画像変更ボタンへのフォーカスが取れる', async () => {
  // ...
});
```

| フィールド | 必須 | 内容 |
|---|---|---|
| `検証内容` | 必須 | 確認する内容 |
| `検証手順` | 必須 | 何をどの順番で実行するか |
| `合格基準` | 必須 | 合格判定基準 |
| `手動で確認する理由` | 必須 | なぜ自動・Agenticでなく手動なのか |
| `実施タイミング` | 任意 | いつ実施するか |

## エージェントの利用方法

### Agentic Verificationの自律実行

エージェントは`VERIFY:agentic`をgrepで抽出し、`検証手順`にしたがって自律的に検証します。`合格基準`を満たしたか判定し、結果を報告します。

```bash
grep -rn "VERIFY:agentic" tests/ src/ --include="*.test.ts" --include="*.spec.ts"
```

`自動テスト化する条件`が満たされていれば、自動テストへの昇格を提案します。

### 手動検証のチェックリスト提示

エージェントは`VERIFY:manual`をgrepで抽出し、`検証手順`と`合格基準`をユーザーに提示します。ユーザーは手順にしたがって実機で確認し、結果を返します。

```bash
grep -rn "VERIFY:manual" tests/ src/ --include="*.test.ts" --include="*.spec.ts"
```

リリース前のチェックリスト生成では、`実施タイミング`が「リリース前」のものを優先的に抽出します。

## 完全な例

```ts
// npm run test:run -- src/features/profile/avatar.spec.ts

// VERIFY:meta
//   対象機能:           プロフィール画像変更
//   想定ユーザー:       ログイン済みのエンドユーザー
//   目的:               プロフィール画像を新しい画像に置き換える
//   前提条件:           ログイン済み / 既存画像あり
//   起こり得る外部条件: cond.network (オフライン) / cond.external (ストレージAPI 5xx)
//   テスト範囲外:       内部APIの詳細 / モデレーション判定
//   最終レビュー日:     2026-05-13

import { test, expect } from '@playwright/test';

test.describe('プロフィール画像変更', () => {
  test('画像を選択して保存できる', async ({ page }) => {
    await page.goto('/profile');
    await page.getByRole('button', { name: '画像を変更' }).click();
    await page.setInputFiles('input[type="file"]', 'fixtures/avatar.png');
    await page.getByRole('button', { name: '保存' }).click();
    await expect(page.getByRole('status')).toHaveText('保存しました');
  });

  // VERIFY:agentic
  //   検証内容:               border-radiusがdesign tokenの値で適用されている
  //   検証手順:               1. ブラウザMCPでプロフィール画面を開く 2. avatar要素を取得 3. getComputedStyle().borderRadius
  //   合格基準:               計算値が完全な円形になる値
  //   自動テスト化しない理由: visual regressionは誤差吸収のため数値が見えない
  //   自動テスト化する条件:   ハードコード値が再発したらe2eに昇格

  // VERIFY:manual
  //   検証内容:           スクリーンリーダーでの読み上げ順序の自然さ
  //   検証手順:           1. VoiceOver有効化 2. プロフィール画面遷移 3. VO+矢印で読み上げ
  //   合格基準:           「プロフィール画像」→「画像を変更」→「氏名入力」の順
  //   手動で確認する理由: axeでは読み順の意味的な自然さは検出できない
  //   実施タイミング:     リリース前
  test('画像変更導線がアクセシブル', async ({ page }) => {
    // ...
  });
});
```

## 落とし穴

| 落とし穴 | 対処 |
|---|---|
| 必須フィールドを空にする | パース失敗で警告が出ます。書くべきものがなければそのVERIFYブロックごと削除します。 |
| 検証手順に「実値を確認する」とだけ書く | エージェントが自律実行できません。「どの要素を / どのAPIで / 何の値を取る」まで具体的に書きます。 |
| metaを書かずにagentic / manualだけ書く | UCファイルが消えるとテストの文脈が失われます。metaは必須です。 |
| 1ブロックの中に複数の検証内容を混ぜる | 1ブロック1検証が原則です。複数あればVERIFYブロックを分けます。 |
| 過去のAgentic検証結果をVERIFYコメントに書く | コメントは「何を検証すべきか」の仕様です。実行ログは別途残します。 |
