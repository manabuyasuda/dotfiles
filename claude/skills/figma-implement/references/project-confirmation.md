# ユーザーへの提案と承諾の進め方

各Stepの判断結果をユーザーに提案して承諾を得る際の、共通の質問・確認ルールです。各referenceの「ユーザーへの提案と承諾を得る」セクションからこのファイルを参照します。

## 質問・確認のルール

- 確証がある判断はまとめて提示する。質問の形にしない
- 不明点・複数案がある箇所は`AskUserQuestion`で確認する。1回の`AskUserQuestion`には質問を1つだけ入れ、必ず1問ずつ尋ねる
- 質問文は「はい」「いいえ」で答えられる肯定文にする。推奨案が「はい」になるよう組み立てる（否定文は理解しにくく、計画を前進させる方向に答えにくいため）
- 各選択肢の`description`には、その選択がもたらす意味やトレードオフを簡潔に書く
- 抽象度の高い項目から始め、回答を踏まえて具体的な項目に絞り込む
- 質問文は複合名詞を避け、助詞（の・による・に関する等）で修飾関係を明示する。主語を省略しない
- すべての分岐が解消されたら、合意内容をまとめて提示してから次のStepに進む

## 抽象→具体の絞り込み順

抽象度の高い項目から先に質問し、回答に応じて具体的な項目を絞り込みます。各Stepで何が抽象・具体に当たるかは、参照元のreferenceに記載されています。

順序は「抽象（採用方式・分類・全体方針）」から「具体（配置場所・命名・属性・パラメーター）」へと進みます。

逆順で具体から聞くと、あとで抽象方針が変わったときに具体の合意がすべてやり直しになります。

## 合意内容の提示形式

すべての確認が終わったら、以下の形式で合意内容をまとめて提示してから次のStepに進みます。

```markdown
**合意内容**

- {項目1}: {決定内容}
- {項目2}: {決定内容}
- {項目3}: {決定内容}

この内容で実装に進みます。
```

見出しは`##`ではなく太字（`**`）にして、親プロセスのMarkdown階層と衝突しないようにします。合意内容の各項目は、各referenceの「提案で必ず触れる項目」表に対応します。

## 早めの相談

最終提案を待たずに、以下のタイミングでは途中でも相談してください。

- 自律推測の確証がない
- 関連実装の確認で迷いが残った
- 判断材料が不足している

誤った方向に進むと、認識合わせがやり直しになり手戻りが大きくなります。
