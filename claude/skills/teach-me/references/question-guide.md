# AskUserQuestionの使い方

`AskUserQuestion`は決定や分岐が発生する節目に絞って使います。グリッドのように一定間隔で挟みません。`grill-me`と異なり連続使用はせず、質問と質問の間には必ず実作業(調査・実行・執筆)を挟みます。

短いトピックでもフェーズ境界と章境界の質問は省略しません。1セッションでの合計回数は短いトピックで6〜9回、中規模で10〜13回、大規模で15回以上を目安にします。

## 節目の3階層

| 階層 | 出現タイミング | 1セッションあたりの回数 |
|---|---|---|
| フェーズ境界 | 流れのステップ間(目的合意→計画→解説→改変→完了) | 4〜6回 |
| 章境界 | 解説フェーズ内で1章書き終えた直後 | 3〜7回 |
| 例外的な確認 | 前提知識が分かれそうな箇所・危険操作の直前 | 必要に応じて |

サブセクション境界では原則として使いません。1章の中で何度も質問すると流れが切れます。

## フェーズ境界の質問の型

| フェーズ | 想定される質問 |
|---|---|
| 目的合意 | 何のためにこれを学びたいですか(機能追加 / 採用判断 / 不具合調査 / チーム共有) |
| 目的合意 | 完了条件として何を達成できれば終わりにできますか |
| 目的合意 | 今回深追いしない領域はどこですか |
| ファイル作成前 | 出力先と粒度はどうしますか(`docs/learn/`既定 / 別パス / 浅め / 深め) |
| 計画提示後 | この章立てで進めますか、それとも調整しますか |
| 改変フェーズ入口 | 小さな改変実験まで進めますか、ここで終わりにしますか |
| 完了判定 | 完了条件はすべて満たせていますか、未達のものはどうしますか |

## 章境界の質問の型

1章書き終えるごとに1個。内容に応じて以下の型から選びます。

| 型 | 質問例 |
|---|---|
| 方向選択 | 次は「動作確認」と「主要コンポーネント」どちらに進みますか |
| 深掘り判定 | 今の章でもう少し詳しく見たい箇所はありますか |
| 理解度確認 | ここまでで腑に落ちていない部分はありますか |
| 粒度調整 | 残りの章は今と同じ粒度でよいですか、もう少し速度を上げますか |
| 完了条件照合 | この章で完了条件◯は達成できたと見てよいですか |

## 例外的に挟む質問

| 場面 | 対応 |
|---|---|
| 前提知識の確認 | 「〇〇は既知ですか」と確認してから説明粒度を調整します |
| 危険操作の承認 | リポジトリ外への書き込み・外部API呼び出し・破壊的コマンドの実行前に「〇〇を実行してよいですか」と確認します |

## 選択肢の書き方

各選択肢の`description`には、その選択で得られる内容や所要時間の目安を書きます。「進む」「戻る」のような抽象的なラベルだけにせず、選んだ後に何が起きるかが想像できる粒度にします。
