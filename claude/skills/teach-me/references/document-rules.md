# ドキュメントルール

`teach-me`スキルが生成するMarkdownドキュメントのルールです。

## 成果物の基本方針

- 解説は会話だけで完結させず、必ずMarkdownファイルを成長させながら進めます
- 出力先の既定は`docs/learn/<topic>.md`。リポジトリ外で学習する場合や、ユーザーが別パスを希望した場合のみ変更します
- 章ごとに`Edit`で追記し、その都度ユーザーに該当章を読んでもらってから次へ進みます
- 構造は「目次 → 全体像 → 各章 → まとめ → 次の一歩」を骨格にします
- 章の見出しは固定の階層を保ち、後から目次として再生成しやすくします
- 認識論的ステータス(静的事実・実行で確認・推測)を文中で明示します(例:「(コードより)」「(実行で確認)」「(推測)」)

## コードブロックの表記規約

GitHubとZennの両方で読みやすくするため、ファイルパスを冗長に書きます。

````markdown
```ts title="src/foo.ts:42-58"
// src/foo.ts:42-58
export function foo() { ... }
```
````

- 言語指定で構文ハイライトを効かせます
- `title`属性はZenn等のキャプション対応環境で表示されます
- 本文1行目のパスコメントは`title`が無視される環境(GitHub含む)でも情報を残します
- 実行結果は`bash`ブロックで「コマンド + 出力 + 1行の解釈」をセットにします

## 図表の選び方

| 表現したいもの | 手段 |
|---|---|
| 階層・包含関係 | 箇条書きまたはMermaid `graph TD` |
| データフロー・依存関係 | Mermaid `graph LR` |
| 状態遷移 | Mermaid `stateDiagram-v2` |
| 時系列・呼び出し順序 | Mermaid `sequenceDiagram` |
| ファイル構造 | ASCIIツリー(`tree`コマンドの出力をベースに整形) |
| 概念の対比・トレードオフ | Markdownテーブル |
| 実UIの挙動・レンダリング結果 | Playwright MCP経由のスクリーンショット |
| Mermaidで表現しきれない自由なレイアウト | SVGをインラインで直接書く |

Playwright MCPのスクリーンショットは`![説明](./<topic>.assets/xx.png)`で埋め込みます。

### SVGをインラインで書くときの規約

- `<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 W H">`で開始し、`width`/`height`属性は付けません(レスポンシブに表示させるため)
- 色はCSSの変数またはGitHubダーク/ライトの両方で読める色を選びます(`#666`・`currentColor`など)
- テキストは`<text>`要素で配置し、`font-family`は指定しません(レンダラー既定に任せるため)
- 図が複雑になる場合は`<g>`でグループ化し、コメントで構成要素を区切ります
- 100行を超えるSVGは別ファイル(`<topic>.assets/xx.svg`)に切り出し、本文からは`![](./...)`で参照します

## 記号・マークアップのルール

- 太字(`**`)は使いません。強調が必要な箇所は文の構造や語順で伝えます
- 水平線(`---`)を視覚的な区切りとして使いません。セクションの区切りは見出しで表現します
- リスト項目の先頭に絵文字をラベルとして付けません
- コードブロック(` ``` `)は、ソースコード・コマンド・設定ファイル・実行結果など機械が解釈する内容や、そのまま端末に貼り付ける内容にのみ使います。手順や説明文を装飾する目的では使いません。ユーザーがそのままコピーする出力テンプレートは例外として使ってかまいません
- 引用記法(`>`)は他者の発言や外部資料からの引用にのみ使います。前提条件や注意書きを目立たせたい場合はGitHub Flavored Markdownのalert記法(`> [!NOTE]`など)を使います
- コロン(`:`)で説明やブロック要素を導きません。短い項目と説明の対は表を使い、各項目に段落が必要な場合は見出しと本文で構造化します
- h1からh6まで意味的な階層関係に沿って使用します

## 日本語表現のルール

- 英数字・記号と日本語の間に半角スペースを入れません。バッククォートで囲んだインラインコードの前後も同様です(正:「Claude Codeを使う」「`true`の場合」、誤:「Claude Codeを使う」「`true` の場合」)
<!-- textlint-disable @textlint-ja/ai-writing/ai-tech-writing-guideline -->

- 冗長な表現を短くします(「できます」→「できます」、「する必要があります」→「してください」、「最初に」→「まず」)

<!-- textlint-enable @textlint-ja/ai-writing/ai-tech-writing-guideline -->
- 文は述語で終え、名詞止めにしません
- 漢語の複合名詞は動詞句に置き換えます(「未取得」→「取得できていない」、「定義外」→「対応していない」)
- 名詞を直接つなげる複合名詞を避け、助詞(の・による・に関するなど)または活用形を補って修飾関係を明示します
- 「場合」(条件・仮定的な状況)と「とき」(時間的な局面・瞬間)を使い分けます
- 文末はです・ます調に統一します

## 文章構造のルール

<!-- textlint-disable @textlint-ja/ai-writing/no-ai-hype-expressions -->
- 誇張表現・ハイプ表現を使いません(「革命的な」「ゲームチェンジャー」「究極の」「完全に解決」「パラダイムシフト」など)
<!-- textlint-enable @textlint-ja/ai-writing/no-ai-hype-expressions -->
- 評価を上乗せするだけで実質情報を加えない表現は具体的な事実で代替します
- 根拠のない楽観的な結語は削除します(「今後の発展が期待されます」など)
- 「本質的には」「真に重要なのは」「核心にあるのは」のような深みを演出するだけのフレーズは削除します
- 調査や執筆の経緯は書きません。現時点で正しい結論だけを書きます
- これから何をするかを宣言するだけの前置き文は削除し、内容に直接入ります
- 出典のない権威付けは著者・機関・文書・日付を明記するか削除します
<!-- textlint-disable @textlint-ja/ai-writing/ai-tech-writing-guideline -->

- 主語が不明な受動態を避け、動作の主体を明示します(「処理が行われます」→「システムが処理します」)

<!-- textlint-enable @textlint-ja/ai-writing/ai-tech-writing-guideline -->
- 抽象的な形容より定量的・具体的に書きます(「高速な」→「50ms未満の」)
- 1段落では1つの論点を扱い、論点が切り替わったら段落を分けます
<!-- textlint-disable ja-technical-writing/max-ten -->
- 「なお、」「また、」「さらに、」「加えて、」のような接続詞が複数の文頭に連続する場合、後続の文を接続詞なしで書き直すか、前の文に統合します
<!-- textlint-enable ja-technical-writing/max-ten -->
- 見出し直後の文で見出しを言い換えるだけの記述は削除します
- 同一の対象には同一の用語を使います

## 引用フォーマット

外部資料を引用するときは以下の形式で記載します。

```
＜英語の引用＞
＜日本語訳(日本語を引用する場合は省略)＞
＜[引用した情報のタイトル](URL)＞
```
