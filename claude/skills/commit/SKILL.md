# Commit Skill

Gitコミットを「適切な粒度」で「人とAIに有用なメッセージ」を付けて作成するスキルです。履歴を読む人（自分・他人・AI）にとって価値のある情報だけを残します。

## 原則

### 1. コミットメッセージには「何を」と「なぜ」だけを書く

5W1Hをコミットの情報源に当てはめて、メッセージに書く情報を絞り込みます。

| 項目 | 情報源 | メッセージに書くか |
|------|--------|--------------------|
| When（いつ） | コミットのタイムスタンプ | 書かない |
| Where（どこで） | ファイル名・ディレクトリ | 書かない |
| Who（誰が） | author | 書かない |
| What（何を） | Subject / Body | 書く |
| Why（なぜ） | Body | 書く |
| How（どうやって） | コード / diff | 書かない |

diffを読めばわかることはメッセージに書きません。diffには現れない「動機」「背景」「判断のトレードオフ」を書きます。

### 2. 1コミット = 1論理的変更

分割する基準は以下の通りです。

- 異なる目的の変更が混在している（例: バグ修正 + 無関係なリファクタ）
- 「あの変更だけ戻したい」と言われたときに別々にrevertできない

分割しない基準は以下の通りです。

- 1つの目的を達成するために必要な複数ファイルの変更
- 実装とそれを検証するテスト
- 型定義とその使用箇所

### 3. 承認なしにgit commitしない

コミット履歴の修正は煩雑で、プッシュ後は書き換えられません。このスキルは「計画提案 → 承認 → 実行」の3段階で進み、メッセージ全文を実行前にユーザーが確認できるようにします。

## ワークフロー

### Step 1: 現状を把握する

以下をすべて実行して、変更内容と履歴の文脈を把握します。

```bash
git status
git diff
git diff --cached
git log -n 20 --oneline
```

- diffは省略せずに全部読みます（読まずに推測しません）
- 直近のコミットからメッセージの書式・言語・typeの使い方を読み取ります
- stagingエリアの現状は気にしません。staged / unstagedに関わらずすべての変更をまとめて把握し、実行時に計画通りに改めてステージングします

### Step 2: プロジェクト規約を確認する

以下が存在する場合、必ず読んで従います。

- `commitlint.config.{ts,js,cjs,mjs}` / `.commitlintrc.*` / `package.json`の`commitlint`フィールド
- `.gitmessage` / commitテンプレート
- `CONTRIBUTING.md`のコミット関連セクション
- リポジトリルートおよびサブディレクトリの`CLAUDE.md`

規約がない場合は[Conventional Commits v1.0.0](https://www.conventionalcommits.org/ja/v1.0.0/)を使います。加えて、`git log`の直近20件から言語（日本語／英語）と既存のtype / scopeの使い方を読み取り、Conventional Commitsと矛盾しない範囲で揃えます。

### Step 3: 「なぜ」の情報を集める

diffからは「何を」しか読み取れません。「なぜ」を確定させるため、以下の情報源を順に確認します。

1. 現在の会話履歴 — ユーザーが直前まで取り組んでいた問題、議論の流れ、明示された目的・制約を確認します
2. ブランチ名 — `feature/<id>` / `fix/<bug>`などから関連Issue / PRの番号を推測します
3. 関連するIssue / PR — 番号がブランチ名や直近のコミット、コードのコメントから分かる場合は`gh issue view <N>` / `gh pr view <N>`で本文・コメントを取得します
4. diff内のコメントやTODO — `TODO:` / `FIXME:`の解消が見えるかを確認します
5. 過去のコミット — 同領域の`git log -- <path>`で文脈を確認します
6. 関連するコード — diffだけでは背景が読めないとき、呼び出し元・型定義・テストなど周辺コードを読んで理解を深めます

これらを総合しても「なぜ」が確定しない場合に限り、計画提案時にユーザーに背景を質問します。

### Step 4: コミット計画を提案する

以下の形式でコミット計画を提示し、ユーザーの承認を得てから実行に移ります。メッセージはSubject + Body + Footerの全文を示します。

````
## コミット計画

### Commit 1

**含めるファイル**:
- path/to/file-a.ts
- path/to/file-b.ts

**メッセージ**:

```
<type>(<scope>): <subject>

<body — 「なぜ」を含む。話題の切れ目以外で改行しない>

<任意の Footer 行（Closes #N など）>
Co-authored-by: Claude <noreply@anthropic.com>
```

### Commit 2（分割がある場合）

**含めるファイル**:
- path/to/file-c.ts

**メッセージ**:

```
<type>(<scope>): <subject>

<body>

Co-authored-by: Claude <noreply@anthropic.com>
```

この計画でコミットしてよいですか？修正点があれば指示してください。
````

提案時の注意点は以下の通りです。

- `git status`に出たファイルはすべて計画の対象です（新規ファイルの`??`も含む）
- メッセージは実際にコミットされる形そのままで書きます。プレースホルダー（`<...>`や`TODO:`）を残さないでください

### Step 5: 承認後、計画通りに実行する

承認を得たら、計画された全コミットを連続で一括実行します。各コミットの手順は以下の通りです。

1. インデックスを計画に合わせます。`git reset`（ワーキングツリーには影響しないunstageのみ）で既存のステージング状態を一度クリアし、`git add -A -- <files>`で計画通りのファイルだけをステージングします。`-A`を使うのは`git mv`によるリネームが含まれる場合でも削除側を取りこぼさないためです
2. `git commit -m "..."`または`git commit -F <tmpfile>`で実行します

全コミットを一括実行し、途中でコミットごとに確認を取りません。エラーが発生した場合のみ中断します。

完了後は`git log -n <作成したコミット数> --oneline`の結果を提示します。エラーで中断した場合は、完了済み・失敗・未着手を区別して報告します。

#### エラー時の挙動

| 状況 | 対応 |
|------|------|
| pre-commit hookによる自動修正 | 受け入れて続行します。hookが変更した差分を`git add`で再ステージし、同じメッセージで再コミットします。確認は不要です |
| commitlint違反 | エラー出力を読んでメッセージを修正し、1回だけ再試行します。2回目も失敗したらエラー内容とメッセージ案をユーザーに伝え、判断を仰ぎます |
| その他のエラー（hook失敗、テスト失敗など） | 止まります。状況を報告してユーザーの判断を仰ぎます |

`--no-verify`は使いません。hookは保護機能であり、バイパスは問題の先送りにしかなりません。

## コミットメッセージの書き方

### Subject（1行目）

commitlint設定があればそれに従います（typeの集合、scopeの必須性、最大文字数など）。設定がない場合はConventional Commits v1.0.0に従います。

- 形式は`<type>(<scope>): <description>`です
- 破壊的変更は`!`を付けます（例: `feat(api)!: drop legacy auth`）。またはFooterに`BREAKING CHANGE: <内容>`を書きます
- 末尾にピリオドを付けません
- 命令形・現在形で書きます。日本語の場合はgit logの既存スタイル（体言止め or 「〜する」）に合わせ、履歴がなければ体言止めを使います

一般的なtypeは以下の通りです。

| type | 用途 |
|------|------|
| `feat` | 新しい機能を追加します（fix・refactorに当てはまらない新規追加） |
| `fix` | 既存の不具合を修正します |
| `refactor` | 振る舞いを変えずにコードを整理します |
| `perf` | 処理速度・メモリなどパフォーマンスを改善します |
| `docs` | READMEやコメントなど、ドキュメントのみを変更します |
| `test` | テストコードを追加・修正します（プロダクションコードは変えません） |
| `style` | インデントや空白など、フォーマットのみを変更します（動作は変えません） |
| `build` | ビルドシステムや依存パッケージを変更します |
| `ci` | CI/CDパイプラインの設定を変更します |
| `chore` | 上記のいずれにも当てはまらない変更に使います |

### scopeの決め方

scopeは常に付けます。任意にすると判断によって揺れが生じるため、例外なく機械的に付けます。commitlintでscopeの集合が定義されている場合はそれを優先し、定義されていない場合は以下のルールで決定します。

1. 変更ファイルすべてが属する共通のディレクトリを特定します
2. `src/`, `app/`, `lib/`, `packages/<name>/`などの慣習的なトップレベルディレクトリを除きます
3. 残ったパスの最上位ディレクトリ名をscopeにします
4. リポジトリルート直下のファイルのみの場合は、ファイル名（拡張子を除きkebab-case化）をscopeにします
5. 共通パスがない（複数領域にまたがる）場合は目的が異なる可能性が高いです。「1コミット = 1論理的変更」に従い分割を検討します

例は以下の通りです。

| 変更ファイル | scope |
|---|---|
| `src/components/Button.tsx`, `src/components/Modal.tsx` | `components` |
| `src/utils/date.ts` | `utils` |
| `packages/ui/src/Button.tsx` | `ui` |
| `app/api/users/route.ts` | `api` |
| `package.json`のみ | `package` |
| `README.md`のみ | `readme` |
| `.github/workflows/ci.yml` | `ci` |

固定マッピング（機械ルールより優先）は以下の通りです。

- `.github/`配下 → `ci`

### Body（2行目以降）

Bodyは必須です。変更が小さく見えても「なぜ」は必ず書きます。diffを読めばわかることは書かず、diffには現れない動機や判断を書きます。

書く内容は以下の通りです。

- なぜこの変更が必要だったか（背景・動機・解決する問題）
- 自明でない設計判断・トレードオフ
- 副作用、後方互換性の注意、移行手順
- 作業中にしか知り得なかった情報（調査でわかったこと・断念した選択肢・試して失敗したアプローチなど）

書かない内容は以下の通りです。

- diffを読めばわかること（「foo.tsを編集」「変数XをYにリネーム」）。diffに現れない根本原因・判断の背景・ユーザーへの影響を書きます
- 時刻・作者・ファイル名
- 「動作確認しました」「テストを追加しました」のような情報量ゼロの定型句

Subject行の後、1行空けてから記述します。改行は段落（話題）が切り替わるときだけ入れます。エディタや端末の表示幅に合わせて途中で折り返すことはしません。1段落 = 1文の場合、改行はゼロです。

補足が薄く見える変更でも、Bodyに動機を書きます。例は以下の通りです。

| Subject | Bodyの例 |
|---|---|
| `chore(deps): reactを18.3.0に更新` | セキュリティパッチを取り込みます。破壊的変更はありません。 |
| `docs(readme): typoを修正` | `recieve`を`receive`に修正しました。検索でヒットしやすくなります。 |
| `style(components): フォーマット適用` | Prettierの設定を変更した後、適用していなかったファイルを揃えます。挙動への影響はありません。 |

### Footer

Bodyの後、1行空けてから記述します（Conventional CommitsのFooter規約）。

毎コミット必ず付けるものは以下の通りです。

- `Co-authored-by: Claude <noreply@anthropic.com>` — Claude Codeが支援したコミットを識別します。GitHub上では共著者として表示されます

任意で付けるものは以下の通りです。

- `Closes #123` / `Refs #456` — Issue / PRへの参照
- `BREAKING CHANGE: <内容>` — 破壊的変更の詳細
- 追加の`Co-authored-by:` — ペア作業者など

### 言語

デフォルトは日本語です。直近20件のコミットメッセージのdescription / bodyがすべて英語のみで構成されている場合に限り英語を使います。それ以外（混在 / 履歴なし / 判断できない場合）はすべて日本語です。1コミット内で日英を混在させません。

Conventional Commitsのtype / scopeはdescriptionの言語に関わらず英語のままにします。

### 文体・表記

日本語で書く場合は以下のルールを守ります。SubjectよりBodyへの適用が主です。

- 英数字・記号と日本語の間に半角スペースを入れません。バッククォートで囲まれたインラインコードの前後も同様（正: 「`true`の場合」、誤: 「`true` の場合」）
- 文末はです・ます調で統一します（「〜した」「〜する」「〜している」は「〜しました」「〜します」「〜しています」に修正します）
- 文は述語（動詞・形容詞・助動詞）で終えます。名詞だけで文を終わらせると時制や意図が曖昧になります
- 「未〜」「〜外」「〜値」などの漢語複合名詞は動詞句に置き換えます（「未取得」→「取得できていない」、「定義外」→「対応していない」、「未知値」→「対応していない値」）
- 名詞を名詞に直接重ねる複合名詞は使わず、助詞（の・による・に関する等）や活用形（した・する・な等）を補って修飾関係を明示します
- 「場合」と「とき」を使い分けます：条件・仮定的な状況には「場合」、時間的な瞬間・局面には「とき」
- 動作主体が読み取れない受動態は避け、主体を明示します（「処理が行われます」→「システムが処理します」）
- 冗長な表現は短くします（「することができます」→「できます」、「する必要があります」→「します」）
- 抽象的な形容より定量・具体的な表現を使います。数値を出せる場合は出し、出せない場合は抽象表現のままでかまいません（例: 「大幅に向上」→「従来比200%向上」）
- 仕様・RFC・issueなど根拠となる外部情報はURLやissue番号でFooterに記載します

## 例

### 悪い例

```
update files

src/auth.tsとsrc/session.tsを編集しました。
validateToken関数を修正して、handleRefreshも追加。
動作確認済みです。
```

typeがありません。Subjectに「何を」も「なぜ」もありません。Bodyはdiffの言い換えで動機がありません。「動作確認済み」はコミットの前提なので情報量がありません。どのような動作確認をしたかを書いて初めて意味を持ちます。

### 良い例

```
fix(auth): 期限切れリフレッシュトークン時のクラッシュを修正

セッション復元時にrefresh tokenが期限切れだった場合、catchされない例外でアプリが落ちていました。匿名ユーザーにフォールバックすることで、ログイン画面へ正常に誘導できます。

Closes #1234

Co-authored-by: Claude <noreply@anthropic.com>
```

typeとscopeで正しく分類しています。Subjectが「何を」を端的に表しています。Bodyが「なぜ」と「判断」を説明しています。diffには現れない情報だけを書いています。

## してはいけないこと

- ユーザーの承認なしに`git commit`を実行しません
- `--no-verify`を付けてhookをスキップしません
- 依頼されていないのに`git push`しません
- 依頼されていないのに`git commit --amend` / `git rebase`で履歴を書き換えません
- diffを読まずに推測でメッセージを書きません
