# npmパッケージのセキュリティ検証

## このスキルが扱う範囲

「あるnpmパッケージを今このプロジェクトに入れて／更新してよいか」を、5つの観点で検証してGO / HOLD / NO-GOの3段階で判定して返します。

- 対象は新規追加と既存パッケージのアップデートの両方を含みます
- 検証粒度は対象パッケージ1個ごととし、複数指定された場合は1個ずつ検証して1つの表に集約します
- ユーザーのプロジェクトへの書き込みは原則行いません。例外は次の2つに限ります
  - 隔離tmpディレクトリでの検証用install（観点1・観点2で利用）
  - ユーザー承認を得た上でのlockfile生成（ベースライン取得用）
- プロジェクト全体の依存ツリースキャン・継続監視・代替パッケージ提示は対象外とします

## 動作検証ガイド

スキル本体または参照ファイルを変更したときは、`VERIFICATION.md`（判定ロジックの回帰検証）で代表パッケージの判定が壊れていないかを確認してください。期待判定が事前にわかっている代表パッケージで5観点と総合判定が正しく出るかを実機で照合する形式です。

検証種別は判定ロジック以外にも複数あります（セキュリティ設計・対話フロー・エッジケース）。本ファイルが現状カバーしている範囲とカバーしていない範囲は`VERIFICATION.md`冒頭の表で整理しています。種別が増えた場合は`verifications/`ディレクトリ配下に分割される予定です。

## パッケージマネージャー固有の操作は外部ファイル参照（必須ルール）

パッケージマネージャー固有の手順（install / update / audit / lockfile生成 / advisory抽出 / peer現状値取得 / 設定実効値取得）は、本ファイルには記載しません。以下の参照ファイルに集約してあります。

| パッケージマネージャー | 参照ファイル |
|---|---|
| npm | `package-managers/npm.md` |
| pnpm | `package-managers/pnpm.md` |
| yarn v1 (Classic) | `package-managers/yarn-classic.md` |
| yarn v2+ (Berry) | `package-managers/yarn-berry.md` |

このスキルを実行するエージェントは、以下のルールを必ず守ります。

- Step 3でパッケージマネージャーを検出したら、対応する参照ファイルを`Read`ツールで必ず読み込みます
- 以降のStep（とくにStep 2-2 / Step 3 / Step 5 / Step 6観点1 / Step 6観点5）で必要なコマンドや抽出式は、参照ファイルに記載されているものをそのまま使います
- 参照ファイルに記載のない手順を独自に考案して、正常系で実行してはいけません（自己流の手順は判定精度を落とします）
- パッケージマネージャーを検出できない場合は中断し、ユーザーに確認します（推測で進めません）

問題発生時のみ、以下の例外として自律対応が許されます。

- 参照ファイル記載のコマンドがエラーで失敗したとき
- 参照ファイル記載の抽出式が想定外の出力（空・null・想定外のキー）を返したとき
- 参照ファイル記載の前提（lockfile存在・node_modules存在など）が満たされていないとき

これらの問題が発生した場合に限り、エージェントは自律的に対応してかまいません。対応の例は、フォールバック経路への切り替え、ユーザーへの確認・通知、判定表への「取得失敗」明記、`AskUserQuestion`での代替手順の提案などです。自律対応した場合は、判定詳細欄にその旨と取った代替手段を明記します（後で参照ファイル側に反映できるように）。

## 完了条件

すべての対象パッケージについて、後述の5つの観点を検証し、表形式の判定と推奨アクションをユーザーに提示した状態です。途中で止めず、最後に必ず推奨アクションまで出力します。

## 実行フローの概要

1. 前提確認
2. 設定ファイル検出と実効値の解決
3. パッケージマネージャー検出と参照ファイル読み込み
4. 対象パッケージとバージョン解決
5. 現状ベースライン取得（lockfileがなければユーザーに生成可否を確認）
6. パッケージごとの5つの観点検証
7. 判定
8. 出力
9. オーバーライド対応

## 実行手順

### Step 1: 前提確認

`npm`が利用できることを確認します。

```bash
npm --version
```

`socket`CLIは観点2のサプライチェーンリスク検出に使います。隔離tmpディレクトリで`socket npm install --package-lock-only`を実行する形なのでログインは不要です。

```bash
command -v socket && socket --version
```

`socket`が見つからない場合、観点2を判定できなくなります。そのまま続行すると「4観点+1不明」の検証になるため、`AskUserQuestion`でユーザーに以下の3択を確認します。

| 選択肢 | 挙動 |
|---|---|
| インストールして続行（推奨） | `npm install -g @socketsecurity/cli`を実行してから検証を続ける |
| そのまま続行（観点2は不明扱い） | socketを使わずに残り4観点で判定。判定詳細欄に「観点2: 不明（socket未インストール）」と明記 |
| 中断 | 検証を始めずに終了 |

「インストールして続行」を選んだ場合、次のコマンドを実行します。npmレジストリ上のパッケージ名は`@socketsecurity/cli`で、インストール後のコマンド名が`socket`です。

```bash
npm install -g @socketsecurity/cli
```

このコマンドはユーザーのプロジェクトには触れず、nodenv等の管理下のグローバルbinにバイナリを追加するだけです。完了後、`socket --version`で動作確認してから検証を続行します。インストールが失敗した場合は、失敗内容を伝えて「そのまま続行（不明扱い）」または「中断」のいずれかを再確認します（同じインストールを再試行しても結果が変わる見込みは薄いため、再試行は選択肢から外します）。

### Step 2: 設定ファイルを検出して実効値を解決する

設定の取得は2段階で行います。

#### 2-1. 静的に読む（リポジトリルートのみ）

| ファイル | 読む設定キー | 効く観点 |
|---|---|---|
| `.npmrc`（リポジトリルート） | `legacy-peer-deps`, `strict-peer-dependencies`, `auto-install-peers`, `tag`, `save-prefix`, `save-exact` | peerDeps判定 / バージョン解決 |
| `pnpm-workspace.yaml` | `strict-peer-dependencies`, `auto-install-peers` | peerDeps判定 |
| `.yarnrc.yml` | `nodeLinker`, `packageExtensions` | peerDeps判定 |
| `package.json` | `packageManager`, `overrides`, `resolutions`, `pnpm.overrides` | パッケージマネージャー特定 / バージョン解決 |

#### 2-2. 実効値と取得元を取る

実効値の取り方はパッケージマネージャーごとに異なります。Step 3で検出した参照ファイルの「7. 設定取得方法」セクションに記載されたコマンドのみを使用します。

抽出した設定はStep 6・Step 7・Step 8で使う「設定スナップショット」として保持し、判定の詳細列にも反映します。スナップショットには「キー、実効値、取得元」を併記します。取得元の判別可否はパッケージマネージャーごとに異なるため、参照ファイルの記載にしたがって記録します。

### Step 3: パッケージマネージャーを検出し、参照ファイルを読み込む

リポジトリルートのlockfileと`packageManager`フィールドから、使用中のパッケージマネージャーを1つに確定します。

| 検出ソース | 確定するパッケージマネージャー | 読み込む参照ファイル |
|---|---|---|
| `pnpm-lock.yaml` | pnpm | `package-managers/pnpm.md` |
| `yarn.lock` + `.yarnrc.yml` | yarn v2+ (Berry) | `package-managers/yarn-berry.md` |
| `yarn.lock`のみ | yarn v1 (Classic) | `package-managers/yarn-classic.md` |
| `package-lock.json` | npm | `package-managers/npm.md` |
| いずれもない | （Step 5でユーザーに確認） | （ユーザー回答後に確定） |

確定したら、対応する参照ファイルを`Read`ツールで必ず読み込みます。以降のStep 5 / Step 6観点1 / Step 6観点5では、この参照ファイルに記載されたコマンドと抽出式のみを使用します。

複数lockfileが同居している場合（例: `package-lock.json`と`pnpm-lock.yaml`が両方ある）は、`AskUserQuestion`でどちらを使うか確認します。勝手に決めません。

`devDependency`として入れたい場合のオプション（`--save-dev` / `-D`など）は、参照ファイルの「2. 推奨install / updateコマンド」セクションに記載されています。付けるかどうかをユーザーに確認します。

### Step 4: 対象パッケージとバージョンを解決する

依頼から対象パッケージ名と希望バージョンを取り出します。

| ユーザー指定 | 解決方法 |
|---|---|
| 未指定（例: `reactを入れて`） | Step 2で取得した`tag`の実効値を見る。`tag`が`latest`以外に設定されていればそのタグ、未設定なら`latest`。`npm view <pkg> dist-tags.<tag>`でバージョンを取得 |
| 範囲指定（例: `^18`） | semverライブラリは使わず、`AskUserQuestion`で「latest（または設定された`tag`）を候補として提示し、別バージョンを指定するか確認」する |
| 固定指定（例: `19.0.2`） | そのバージョンを使う |

`package.json`の`overrides` / `resolutions` / `pnpm.overrides`に対象パッケージの記載があれば、「解決バージョンが上書きされる可能性」を判定表の詳細欄に明示します。

複数パッケージが指定された場合は、ここで`[(pkg, version), ...]`の配列に正規化してからStep 6で1個ずつ回します。

### Step 5: 現状ベースラインを取得する

検証対象とは独立に、現時点のプロジェクトの脆弱性件数を取得します。Step 3で読み込んだ参照ファイルの「4. ベースラインauditコマンドと抽出式」セクションに記載されたコマンドのみを使用します。

参照ファイルの抽出式は、結果を`{critical, high, moderate, low}`形に正規化したオブジェクトとして返します。

lockfileが存在しないとき、またはauditコマンドがlockfile不在を理由に失敗したときは、`AskUserQuestion`で「lockfileを生成してベースラインを取りますか？」と確認します。ユーザーが承認したら、参照ファイルの「3. lockfile生成コマンドと副作用」セクションに記載されたコマンドのみを使用します。副作用（`node_modules`を同時に生成するかどうか）も参照ファイルに記載されているので、ユーザー確認時に明示します。

ユーザーが拒否した場合は「ベースライン未取得」と記録して続行します。検証結果には「ベースラインは未取得（lockfile未生成）」と明示します。

### Step 6: 各パッケージで5つの観点を検証する

対象`(pkg, version)`のペアごとに、5つの観点を順番に取ります。

事前に「観点1の経路2（隔離tmpの`npm audit`）」と「観点2のsocketスキャン」で共通利用する隔離tmpディレクトリを1つ用意します。macOSでは`mktemp -t`の引数がプレフィックスそのものとして扱われる（`XXXXXX`がリテラル化される）ため、必ずフルパステンプレートを渡します。

```bash
# 隔離tmpディレクトリの作成（macOS BSD・GNU Linux両対応）
WORK=$(mktemp -d "${TMPDIR:-/tmp}/socket-verify-XXXXXX")
mkdir -p "$WORK/.npm-cache"

# assert: mktempが返したパスがsocket-verify-プレフィックスを含むことを確認します
# （/tmp固定はしません。$TMPDIRが違う場所を指していても許容します）
case "$WORK" in
  */socket-verify-*) ;;
  *) echo "unexpected WORK=$WORK"; exit 1 ;;
esac

# assert通過後にtrapを張ります。これにより観点1の経路2と観点2を通った後、
# シェルが終了する時点で必ず$WORKが削除されます（早期exitや失敗時も拾えます）。
# assertより先にtrapを張ると、不正パスを掴んだまま消しに行くので順序が重要です。
trap 'rm -rf "$WORK"' EXIT

# 全パッケージをまとめて1回でinstallします（socketスキャンも1回で済みます）
# npm側フラグ（多重防御）
#   --package-lock-only : install scriptsを実行しません（メインの防御）
#   --ignore-scripts    : npm版で挙動が変わる場合に備えた多重防御
#   --cache=...         : tarballをtmp内のキャッシュに置き、$WORK削除時に一緒に消します
( cd "$WORK" && socket npm install \
    --package-lock-only \
    --ignore-scripts \
    --cache="$WORK/.npm-cache" \
    <pkg1>@<v1> <pkg2>@<v2> ... ) 2>&1 \
  | tee "$WORK/socket-output.txt"
```

この`$WORK`は観点1の経路2と観点2の両方から参照します。削除は`trap`がスクリプト終了時に拾うため、観点1・観点2の検証中はディレクトリが残ったままです。

##### 検証用installのセキュリティ設計

コードブロックのコメントに記載した多重防御フラグは、サプライチェーン攻撃の主要ベクトルを閉じるためのものです。閉じる攻撃ベクトルと残存リスクをまとめます。

| 攻撃ベクトル | 状態 | 防御している仕組み |
|---|---|---|
| install scripts（preinstall/install/postinstall） | 閉じた | `--package-lock-only`と`--ignore-scripts`の二重防御 |
| パッケージファイルの`require`/`import`連鎖実行 | 閉じた | node_modulesに展開しない |
| tarball展開時のpath traversal | 閉じた | 展開しない |
| ユーザーグローバルキャッシュ汚染 | 閉じた | `--cache=$WORK/.npm-cache`で隔離 |
| tmpディレクトリの永続化 | 閉じた | `trap`で検証終了時に削除 |
| npm本体やsocketの未知の脆弱性 | 残存（低） | 「現時点で既知の攻撃手段はないが、将来見つかる可能性はある」レベルのリスク。ネットワークとファイルシステムを遮断する環境（コンテナー等）に置かない限り残る |

日常的な検証用途では十分なリスク低減ができていますが、「ゼロ」ではありません。事前にマルウェア疑いが強いと分かっているパッケージを意図的に検証する場合は、エッジケース「事前にマルウェア疑いが強いパッケージを検証する場合」を参照してください。

#### 観点1: 既知CVE

観点1は2つの経路で観測します。アップデート時はユーザーのプロジェクトのlockfileに対するaudit、それ以外はStep 6冒頭で作った隔離tmpの`package-lock.json`に対する`npm audit`を使います。

##### 経路1: プロジェクトのauditを使う（アップデート時）

Step 3で読み込んだ参照ファイルの「5. 観点1の対象パッケージadvisory抽出式」セクションに記載された抽出式のみを使用します。auditのJSON形式はパッケージマネージャーごとに違うため、自己流のjqを書くともっとも一般的に使われるnpmで常に空が返るなどの不具合が発生します。

##### 経路2: 隔離tmpの`npm audit`を使う（新規追加時とフォールバック時）

次のいずれかに該当する場合は、Step 6冒頭で作った隔離tmpの`package-lock.json`に対して`npm audit`を走らせます。隔離tmpには`<pkg>@<version>`のlockfileがsocket npmによって作られているため、対象パッケージのadvisoryを正しく取得できます。

- 新規追加時: ユーザーのプロジェクトのlockfileに対象パッケージがないため、プロジェクトのauditでは新パッケージのadvisoryを拾えません
- フォールバック時: アップデート時に経路1の抽出式が空（重大度カウントが全部0など）を返した場合（主にyarn berry 4.xの版揺れ）

```bash
( cd "$WORK" && npm audit --json ) | jq '.vulnerabilities["<pkg>"]'
```

判定詳細欄には「観点1ソース: 経路1（プロジェクトaudit）/ 経路2（隔離tmpのnpm audit）」のいずれを採用したかを明示します。`deprecated`に"security"を含む場合は重大度をcritical相当に引き上げます（観点3でも`deprecated`を扱いますが、"security"含みは観点1にもエスカレーションします）。重大度ごとの最終判定はStep 7の判定表に従います。

#### 観点2: サプライチェーンリスク（socket npm）

Step 6冒頭で作った隔離tmpの`socket-output.txt`を文字列マッチで判定します。socket npmは現状JSON出力に対応していないため、人間可読出力を解釈する暫定実装である点を判定表の詳細欄に明記します。

| socketの出力 | 観点2の評価 |
|---|---|
| `found no risks`を含む | 問題なし |
| `found risks`を含み、`critical` / `malware` / `supply.?chain`（`grep -iE`で空白・ピリオド・キャメルケースを吸収）のキーワードあり | NO-GO相当 |
| `found risks`を含み、`high`のキーワードあり | HOLD相当 |
| 出力に`risks`の言及があるがseverity不明 | HOLD相当（保守的に倒す） |
| `socket`コマンドがない | 不明として扱う（観点2単独ではNO-GOにしない） |
| 隔離tmpでネットワーク等の理由でinstall自体が失敗 | 取得失敗として扱い、ユーザーに通知 |

socket CLIのバージョン更新で出力形が変わった場合、文字列マッチに失敗する可能性があります。判定表の詳細欄にsocketのバージョンを併記しておくと、将来トラブった際の手がかりになります。

##### 複数パッケージ時のsocket出力混在

複数パッケージを1つのtmpにまとめると、socket出力（`socket-output.txt`）が全パッケージ混在になります。リスク言及がない場合は問題ありませんが、`risks`の言及が出た場合は文字列マッチだけではどのパッケージ由来かを切り分けられません。そのときは観点2の判定をいったんHOLDに置き、該当行を引用してユーザーに提示します。さらに厳密に切り分けたい場合は、ユーザーに「パッケージ別のtmpで再検証してよいか」を確認します。

#### 観点3: メンテナンス状況

```bash
npm view <pkg>@<version> --json \
  | jq '{license, time_modified: .time.modified, maintainers_count: (.maintainers | length), repository_url: .repository.url, deprecated, peerDependencies}'
```

`peerDependencies`もここで一緒に取り出します（観点5で利用）。`npm view`を1度叩いてまとめて取得しておくことで、観点3と観点5で重複した通信を避けます。

判定は以下の通りです。

- 最終更新（`time.modified`）が現在から2年を超えていればHOLD。なお`.time.modified`はパッケージ全体の最終publish時刻であり、対象バージョン固有の公開時刻ではありません。詳細欄では「（パッケージ全体の最終更新）」と注記します
- メンテナが1人だけならHOLD
- `repository.url`がないならHOLD
- `deprecated`フィールドが空でない場合は、内容に"security"を含むなら観点1にエスカレーション（critical相当）、それ以外はこの観点でHOLD。`deprecated`シグナルの責務は観点3（メンテナンス）に集約します
- 上記すべてに当てはまらないなら問題なし

メンテナンス観点は単独ではNO-GOにしません。

#### 観点4: ライセンス

`npm view`の`license`を取り、形式ごとに分岐します。

```bash
npm view <pkg>@<version> --json | jq '.license, .licenses'
```

形式別の扱いは以下の通りです。

| 形式 | 抽出方法 |
|---|---|
| 文字列（単一識別子）`"MIT"` | そのまま判定 |
| SPDX複合式（`OR`）`"(MIT OR Apache-2.0)"` | `(`・`)`・空白を除去し`OR`で分割して全トークンを抽出 |
| SPDX複合式（`AND`）`"(MIT AND CC-BY-4.0)"` | `AND`で分割して全トークンを抽出 |
| 旧形式オブジェクト`{type, url}` | `.type`を抽出 |
| 旧形式配列`[{type, url}, ...]` | 各要素の`.type`をすべて抽出 |
| `SEE LICENSE IN ...` | カスタム扱い |
| 未設定 | 「記載なし」扱い |

判定は上から順に適用します（最初に一致したものを採用、それ以降の行は評価しません）。

| 優先順 | 抽出後のトークンの状態 | 評価 |
|---|---|---|
| 1 | トークンに`UNLICENSED`を含む（`OR`／`AND`を問わない） | NO-GO |
| 2 | 全トークンが許容リスト（MIT / Apache-2.0 / BSD-2-Clause / BSD-3-Clause / ISC / 0BSD / Unlicense）に含まれる、または`OR`式で許容リストのトークンが1つ以上含まれる | 問題なし |
| 3 | 上記以外（GPL系／LGPL系／AGPL系／カスタム／記載なし／`AND`式で一部だけ許容） | HOLD |

`UNLICENSED`を最優先でNO-GOに倒すのは、たとえば`(UNLICENSED OR MIT)`のような実在しにくい組み合わせでも安全側に倒すためです。

`UNLICENSED`（全大文字）と`Unlicense`（先頭のみ大文字）はまったく別物です。混同に注意します。

| 文字列 | 種別 | 評価 |
|---|---|---|
| `UNLICENSED` | npm独自の「公開しない」マーカー。プロプライエタリ・社内専用・非公開を示す | NO-GO |
| `Unlicense` | SPDX識別子の1つ。パブリックドメイン献辞に相当する寛容なライセンス（例: tweetnaclが採用） | 許容リストに含めGO |

判定実装では大文字小文字を厳密に区別します。`tolower()`などで丸めると`Unlicense`までNO-GOに倒れるので注意します。

許容リストの拡張をユーザーが希望する場合はオーバーライド対応（Step 9）として扱います。

#### 観点5: peerDependencies

観点3で取った`npm view`結果の`peerDependencies`を、プロジェクトの現状依存と照合します。

##### 現状peerバージョンの取得

Step 3で読み込んだ参照ファイルの「6. 観点5のpeer現状値取得方法」セクションに記載された手順のみを使用します。パッケージマネージャーごとにnode_modulesの有無・lockfile形式が異なるため、自己流の手順は取り違えの原因になります。

参照ファイル記載の方法でも現状peerが取れない場合（lockfile直読み不可・node_modules不在など）は、判定表に「現状peer未取得」と明示してHOLDに倒します。ユーザーに`<pm> install`を促す案内文を添えます。

##### 範囲整合の判定（semverライブラリ不使用）

1. peerの範囲指定（例: `^18.0.0`）から、メジャーバージョンを抽出します（`^` / `~` / 数値プレフィックスから単純取り出し）
2. 現状ロックされているpeerのメジャーバージョンと一致するかを確認します
3. 一致すれば「整合の見込みあり」、不一致なら「不整合の可能性あり」

このメジャーバージョン一致判定は完全とは言えません。判定結果の詳細欄に「peer範囲: ^18 / 現状: 18.3.1 → メジャーバージョン一致」のような形で比較した値を明示し、ユーザーが目視で再確認できるようにします。

##### 設定の実効値を組み合わせて観点5の評価を決める

| peer範囲外か | 設定（実効値） | 観点5の評価 |
|---|---|---|
| 範囲内 / peerなし | — | 問題なし |
| 範囲外 | `auto-install-peers=true`（pnpm） | 問題なし（peerも自動でinstallされる） |
| 範囲外 | `legacy-peer-deps=true`（npm） / `strict-peer-dependencies=false`（pnpm） | HOLD（installは成功するが実行時に壊れる可能性あり） |
| 範囲外 + peerがプロジェクト未インストール | 上記の許容設定なし | HOLD（同時にinstallが必要、または失敗する） |
| 範囲外 | 許容設定なし | HOLD |
| 現状peer未取得（lockfile直読みも失敗） | — | HOLD（情報不足、ユーザーに`<pm> install`を促す） |

判定詳細欄には「peer範囲」「現状バージョン（取得元: node_modules / lockfile / 未取得）」「適用された設定（実効値 + 取得元）」をすべて記載します。

### Step 7: 判定基準を適用する

各観点ごとに以下を機械的に当てはめます。

| 観点 | NO-GO | HOLD | GO |
|---|---|---|---|
| 1. CVE | criticalのadvisoryが当該バージョンに該当 / `deprecated`に"security"を含む（観点3からエスカレーション） | highのadvisoryが当該バージョンに該当 | moderate以下、もしくは該当なし |
| 2. サプライチェーン | socket出力にcritical / malware / `supply.?chain`の言及（観点2判定表の正規表現に同じ） | socket出力にhigh / 不明なriskの言及 | リスク言及なし、またはsocket未インストールで他観点に問題なし |
| 3. メンテナンス | （単独ではNO-GOにしない） | 最終更新が2年超 / メンテナが1人だけ / `repository.url`がない / `deprecated`が空でない（"security"含みは観点1に回す） | 上記いずれにも該当しない |
| 4. ライセンス | トークンに`UNLICENSED`を含む | 抽出トークンに許容リスト外を含む | 抽出トークン全体が許容リストで成立する |
| 5. peerDependencies | （単独ではNO-GOにしない） | 観点5の判定表（Step 6内）でHOLDに該当 / 現状peer未取得 | 設定との組み合わせで「問題なし」 |

総合判定は次の通りです。

- 1観点でもNO-GOがあれば総合NO-GO
- NO-GOがなく、1観点でもHOLDがあれば総合HOLD
- すべてGOなら総合GO

複数パッケージを検証した場合は、もっとも厳しい結果を全体判定とします（1つでもNO-GOなら全体NO-GO、1つでもHOLDなら全体HOLD）。

### Step 8: 出力

下記の固定フォーマットでユーザーに提示します。複数パッケージのときはパッケージごとに表を出し、最後に総合判定と推奨アクションをまとめます。

```markdown
判定: <GO|HOLD|NO-GO> <pkg>@<version>

| 観点 | 結果 | 詳細 |
|---|---|---|
| 1. 既知CVE | <GO|HOLD|NO-GO> | 例: `npm audit`でhigh 1件 / advisory ID xxx（観点1ソース: 経路2） |
| 2. サプライチェーン | <GO|HOLD|NO-GO|不明> | 例: socket npm scanでrisk言及なし / socket CLI v1.1.67 / 文字列マッチによる暫定判定 |
| 3. メンテナンス | <GO|HOLD> | 最終更新2024-11（パッケージ全体）/ メンテナ3名 / repo有 / deprecatedなし |
| 4. ライセンス | <GO|HOLD|NO-GO> | MIT（単一）または`(MIT OR Apache-2.0)` → 全トークン許容 |
| 5. peerDeps | <GO|HOLD> | peer範囲`react@^18` / 現状`react@18.3.1`（取得元: node_modules）/ 設定`legacy-peer-deps=true`（user .npmrc） |

現状ベースライン (<pm> audit): critical=X, high=Y, moderate=Z, low=W
（取得不可の場合: 「ベースライン未取得（lockfile未生成）」と記載）

検証時の設定スナップショット:
- `legacy-peer-deps`: true（取得元: userスコープ。`npm config get --location=user`で確認）
- `tag`: latest（取得元: default。3スコープともdefault値）
- `auto-install-peers`: 未設定
- `package.json` overrides: なし

備考: 取得元の判別はパッケージマネージャーごとに対応状況が異なります。判別可否は各参照ファイルの「7. 設定取得方法」を確認してください。

推奨アクション:
- GOの場合: `<pm> add <pkg>@<version>`を実行してよい
- HOLDの場合: <観点名>の点を確認後にinstallすること。具体的には...
- NO-GOの場合: installしない。理由: <観点名>がNO-GO条件に該当
```

絵文字は使わず、`GO` / `HOLD` / `NO-GO` / `不明`の文字で記載します。

### Step 9: オーバーライド対応

ユーザーがHOLD / NO-GOの判定を理解した上で「警告を承知の上でinstallしたい」と明示した場合のみ、installコマンド文字列を返します。判定表は残し、ユーザーが受け入れた理由を1行添えるよう促します（メッセージ・PR本文・コメントなど、判定をオーバーライドした記録を残す手段はユーザーに委ねます）。

ユーザーから明示の同意がないままNO-GOのパッケージをinstallする提案はしません。

## エッジケースと例外

エッジケースは性質ごとにグルーピングして並べています。「入力に関する例外」「環境に関する例外」「セキュリティに関する例外」「参照ファイルへのポインター」の順です。

### 入力に関する例外

#### 検証対象がプロジェクト内にすでに存在するパッケージ（アップデート系）

`<pm> ls <pkg> --json`で現状のバージョンを取得し、判定表に「現バージョン → 目標バージョン」を併記します。メジャーバージョン跨ぎを観測したときは「破壊的変更の可能性あり」をメンテナンス観点とは別に注記します（判定そのものは変えません）。

#### `npm view`がパッケージ自体を見つけられない

タイポ・スコープミス・公開停止のいずれかです。ユーザーに「該当パッケージがnpmレジストリで見つからない」ことを伝え、Step 4のパッケージ名を再確認します。typosquatの可能性があるため、似た名前の公式パッケージがないかも触れます（ただし代替提案までは深追いしません）。

#### `npm view <pkg>@<version>`で指定バージョンだけが見つからない

パッケージ自体は存在するが、指定したバージョンが該当しないケースです（例: `react@19.99.0`のような存在しない版）。`npm view <pkg>@<version>`は該当版がないと空を返します。これを「パッケージ不在」と同じ経路に流すと案内文がズレるため、別経路として扱います。

経路の見分け方は次の通りです。

```bash
# パッケージ自体の存在確認
if ! npm view <pkg> name >/dev/null 2>&1; then
  echo "パッケージ不在"
elif [ -z "$(npm view <pkg>@<version> version 2>/dev/null)" ]; then
  echo "指定バージョン不在"
else
  echo "存在"
fi
```

「指定バージョン不在」と判定したら、Step 4のバージョン解決に戻り、ユーザーに`npm view <pkg> versions`で取得した公開済みバージョン一覧を提示してから再選択してもらいます。

#### `package.json`の`overrides` / `resolutions`で対象パッケージが上書きされている

Step 4で「上書き対象である」と検出した時点で、判定表に「上書き設定により実際の解決バージョンが変わる可能性」を明示します。ユーザーの期待するバージョンと、実際にプロジェクトで解決されるバージョンが食い違うケースもあるため、両方を提示してユーザーに判断を委ねます。

### 環境に関する例外

#### モノレポでlockfileがワークスペースルートにある場合

`git rev-parse --show-toplevel`で取れる位置にもlockfileがない場合は、`pnpm-workspace.yaml` / `package.json`の`workspaces`フィールドの有無を確認し、ワークスペースルートを推定します。推定できなければStep 3で「lockfileが見つからない、どのパッケージマネージャーをフォールバックとして使うか」をユーザーに確認します。

#### socketがインストールされていない場合

Step 1の前提確認で3択（インストールして続行 / そのまま続行 / 中断）を提示します。詳細はStep 1を参照してください。「そのまま続行」が選ばれた場合は観点2を「不明」として記録し、サプライチェーン観点単独ではNO-GOに倒しません。判定詳細欄に「観点2は不明（socket未インストール、ユーザー選択により続行）」と明記します。

#### Step 6の途中で外部コマンドがネットワークエラーで失敗

`npm view`がネットワーク失敗した場合は、判定を出さずに中断します。「ネットワークエラーだから安全側でGOに倒す」という処理はしません。socketも、ネットワーク失敗と「コマンドなし」を別物として扱い、ネットワーク失敗時は「不明」ではなく「取得失敗」として記録します。

### セキュリティに関する例外

#### 事前にマルウェア疑いが強いパッケージを検証する場合

通常の検証フロー（`--package-lock-only` + `--ignore-scripts` + 隔離tmp）は、未知の悪意パッケージに対する日常的な検証を想定した防御設計です。これに対し、すでに「マルウェアの可能性が高い」と判明している特定パッケージを意図的に検証したい場合（例: typosquat疑いのパッケージ調査、外部から共有された不審なパッケージの精査）は、本スキルだけで完結させるのは推奨しません。

理由は、残存する理論上のリスク（npm本体やsocket本体の未知の脆弱性を突かれる可能性）が、対象パッケージの危険度と釣り合わなくなるためです。詳細はStep 6の「検証用installのセキュリティ設計」を参照してください。

このような場合は以下のいずれかを推奨します。

- Dockerコンテナー内で本スキルを実行します（ホスト環境から完全分離）
- `nsjail` / `firejail`等のサンドボックスツールで実行します（Linux環境のみ）
- 検証専用のVMで実行します
- もしくは検証を断念し、対象パッケージ自体を採用候補から外します

ユーザーから「このパッケージはおそらくマルウェアだが念のため検証して」のような依頼を受けた場合、エージェントは検証を始める前に上記の選択肢を提示し、`AskUserQuestion`で「コンテナー等で実行するか / 本スキルで通常検証するか / 検証を断念するか」を確認します。「通常検証」が選ばれた場合は判定詳細欄に「ユーザー承認のもとで通常検証を実施」と明記します。

### 参照ファイルへのポインター

#### パッケージマネージャー固有のエッジケース

パッケージマネージャー固有のエッジケース（yarn v1のlockfile生成が`node_modules`も生成する件、yarn berryのaudit版揺れ・Plug'n'Play構成など）は、各参照ファイルの「8. このパッケージマネージャー固有のエッジケース」セクションに集約されています。Step 3で読み込んだ参照ファイルの該当セクションを必ず確認します。
