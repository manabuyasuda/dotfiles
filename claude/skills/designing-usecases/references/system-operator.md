# 系統O: 運用者・管理者 × 管理画面 / CLI

## 定義

| 項目 | 内容 |
|---|---|
| 主体 | CS、SRE、社内オペレーター、コンテンツモデレーター。エンドユーザーよりも強い権限を持ち、データを直接操作する。 |
| インターフェース | 管理画面、CLI、SQLコンソール、機能フラグダッシュボード、運用バッチ |
| 達成したいこと | 障害対応、データ修正、機能のロールアウト、権限管理、監査対応 |
| 観察可能性 | DBの状態、監査ログ、影響範囲のプレビュー、通知 |
| 粒度 | 運用業務単位（「違反コンテンツを強制削除する」「特定ユーザーに機能フラグを有効化する」） |
| 典型的な失敗モード | 取り返しのつかない操作の誤実行 / 影響範囲の誤認 / 監査ログが残らない / 権限境界の漏れ |
| 主なテスト種別 | integration / 管理画面 e2e / 権限境界テスト / 監査ログ検証 |

## 観点チェックリスト

### 権限境界

- [ ] 各ロールで実行可能な操作の境界が定義されている
- [ ] ロールごとに「成功 / 403 / そもそもUIに出ない」のいずれかが意図通り
- [ ] 権限昇格（一時的な強い権限の付与・剥奪）の経路が明確
- [ ] 横方向のアクセス（同一ロールでも他テナント/他ユーザーのデータに触れない）

### 監査

- [ ] 操作ログに「誰が / 何を / いつ / なぜ」が記録される
- [ ] 監査ログ自体が改ざんされない（追記のみ、別系統への書き出し）
- [ ] 監査ログのリテンション期間
- [ ] 個人情報を含む操作ログのマスキング
- [ ] ログの検索・追跡が可能（インシデント調査時）

### データ修正

- [ ] マスター更新（システム設定、料金プラン等）
- [ ] 不正データの修復（個別レコード修正）
- [ ] 論理削除と物理削除の区別
- [ ] 一括更新の影響範囲
- [ ] CSV / SQL 経由のバルク操作の整合性

### 機能フラグ・設定変更

- [ ] ON / OFF切替の即時性（キャッシュ、CDNを含む）
- [ ] ロールアウト範囲（特定ユーザー / 割合 / 条件）
- [ ] ロールバックの容易さ
- [ ] 機能フラグ間の依存関係（A=ON が前提で B=ON など）

### 障害対応

- [ ] キューの再投入
- [ ] 失敗ジョブの再実行（冪等性確保）
- [ ] データの巻き戻し（point-in-time recovery）
- [ ] 緊急停止スイッチ（kill switch）
- [ ] 障害時の手動オーバーライド

### 影響範囲の可視性

- [ ] 操作前のドライラン（何件・誰に影響するか）
- [ ] 影響を受けるユーザーへの通知有無
- [ ] 関連サブシステムへの伝播の見通し

### 誤操作の防御

- [ ] 取り返しのつかない操作の確認ステップ（ダイアログ / 文字入力確認）
- [ ] 削除の段階性（論理削除 → 物理削除の猶予期間）
- [ ] 操作の取り消し可能性
- [ ] 「本番 / ステージング」の明確な視覚的区別

### 並行操作

- [ ] 同じ対象を別オペレーターが同時に操作した時の挙動（楽観ロック等）
- [ ] 長時間トランザクションの扱い

## 検証方法への割り当て指針

各観点について、自動テストとAgentic Verificationの両方の選択肢を持ちます。
詳細な使い分けは `references/verification-methods.md` を参照してください。
系統Oに対するAgentic Verificationは副作用が大きいため、必ずステージングまたはread-onlyで実行してください。

| 観点 | 自動テスト | Agentic Verification |
|---|---|---|
| 権限境界 | 権限境界テスト（各ロールで API/UI を叩く） | ステージングで各ロールでログインして実確認 |
| 監査ログ | integration（DB/ログ書き込み確認） | DB MCP で `audit_logs` テーブルを直接 SELECT 検査 |
| データ修正 | integration | ステージングで実操作 → DB MCP で結果確認 |
| 機能フラグ即時性 | integration + e2e | ブラウザMCPで切替操作 → 別タブで反映確認 |
| 障害対応（再実行・冪等性） | integration | ステージングで失敗ジョブを実再投入 |
| 影響範囲のプレビュー | 管理画面 e2e | ブラウザMCPでドライラン実行して結果確認 |
| 誤操作防御の確認ステップ | 管理画面 e2e | ブラウザMCPで確認ダイアログをスクショ |
| 並行操作 | integration（並行リクエスト発火） | 2セッション同時操作（ステージング） |
| キャッシュ無効化 | integration | 実 CDN へのリクエストヘッダ確認、ブラウザMCPで再取得 |

## 系統Oで起きがちな間違い

| 間違い | 対処 |
|---|---|
| エンドユーザーUCに紛れ込ませる | 「管理者でログインしてユーザーをBANする」を系統E扱いにすると、観点（監査ログ・影響範囲・権限境界）が全部抜けます。系統Oとして独立して書いてください。 |
| 権限境界を「権限のあるロールでのテスト」だけで済ます | 権限のないロールで叩いたときに拒否されることを確認するテストが本質です。 |
| 監査ログを軽視する | 監査要件（SOC2等）がある場合、監査ログ欠落は致命的です。観点として最初から含めてください。 |
| 取り返しのつかない操作を1ステップで実行可能にする | ドライラン → 確認 → 実行の段階性を設計に含めてください。 |
| 本番・ステージングの区別がUI上で曖昧 | ヘッダーカラーやバッジで明示し、テストでも視覚的差異を確認してください。 |
