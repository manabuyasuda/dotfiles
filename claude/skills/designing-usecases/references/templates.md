# ユースケース記述テンプレート（系統別）

各系統で記述項目が異なります。1つのユースケースは1つの系統に固定します。

## 系統E テンプレート（エンドユーザー × UI）

```markdown
### UC-E-XXX: <ユースケース名>

- **系統**: E (エンドユーザー × UI)
- **主体**: 例) ログイン済みのエンドユーザー
- **目的**: ユーザーが達成したいこと（プロダクト目的の言葉で書く。技術用語を避ける）
- **前提条件**:
  - 認証状態
  - データ初期状態
  - デバイス・ロケール
- **トリガー**: ユーザー操作 / 通知の受信 / 別画面からの遷移
- **主シナリオ (UI操作)**:
  1. ...画面で...をタップする
  2. ...が表示される
  3. ...
- **代替シナリオ**: 〜のとき: ...
- **異常シナリオ**: 〜が起きたとき、UI上どう見えるか
- **適用される状況の観点**:
  - `cond.external`: HTTPステータス全種、遅延
  - `cond.network`: オフライン遷移
- **観察可能な期待結果**: UI上の表示・遷移・トースト等
- **検証方法の割り当て**:
  - 自動テスト:
    - e2e: 主シナリオ + 1異常シナリオ
    - visual: 完了画面のスナップショット
    - a11y: フォーカス順序とキーボード完遂
  - Agentic Verification:
    - ブラウザMCPで実操作して仕様確定
    - `getComputedStyle()` でCSS実値確認
    - `performance.getEntries*()` で CLS/LCP 実測
- **非対象 (Out of scope)**: 〜は本UCでは扱わない
```

### 系統E のチェックポイント

- 観察可能な期待結果が「UIから見えるもの」だけになっているか（DB状態を期待結果に書いていないか）
- 主シナリオがユーザーの操作言葉になっているか（API呼び出しの記述になっていないか）
- a11y を「適用される状況」ではなく独立した検証項目として扱っているか

## 系統D テンプレート（開発者 × コードAPI）

```markdown
### UC-D-XXX: <APIユースケース名>

- **系統**: D (開発者 × コードAPI)
- **主体**: 例) このコンポーネント/関数を呼び出す開発者
- **目的**: 開発者が実現したい統合（「フォーム送信時に画像をアップロードしたい」等）
- **前提条件**: 依存バージョン / Peer Deps / ランタイム
- **公開シグネチャ**:
  ```ts
  function uploadAvatar(
    file: File,
    options?: UploadOptions
  ): Promise<AvatarResult>
  ```
- **主シナリオ (API呼び出し)**:
  1. 必須引数のみで呼ぶ
  2. resolve した値の型と内容
- **代替シナリオ**:
  - options を渡す
  - 既存値を上書き
  - Suspense境界内で使う
- **異常シナリオ**:
  - 不正引数 → 型エラー（コンパイル時）
  - ネットワーク失敗 → reject の型
  - 中断（AbortSignal）→ reject の型
- **適用される状況の観点**:
  - `cond.external`: レスポンス形状の揺れ
  - `cond.network`: AbortSignal の伝搬
- **観察可能な期待結果**: 戻り値、型、発火するイベント、副作用、ログ
- **検証方法の割り当て**:
  - 自動テスト:
    - unit: 引数バリデーション、resolve/reject
    - 型テスト: シグネチャの全パターン
    - contract: APIレスポンスのスキーマ
    - build smoke: ESM/CJS から import できる
  - Agentic Verification:
    - 最小サンプルを書いて実行・挙動確認
    - `tsc --noEmit` で型エラーを直接確認
    - 実バンドル成果物を検査して tree-shaking 結果確認
- **非対象**: UI 側の表示は別UC (UC-E-XXX) で扱う
```

### 系統D のチェックポイント

- 公開シグネチャが TypeScript で記述されているか
- 型レベルで防げる異常シナリオが「型テスト」になっているか（runtime test に紛れ込んでいないか）
- UI 表示への期待が混入していないか

## 系統O テンプレート（運用者 × 管理画面/CLI）

```markdown
### UC-O-XXX: <運用業務名>

- **系統**: O (運用者 × 管理画面/CLI)
- **主体**: 運用者の役割（CS / SRE / 管理者）と権限ロール
- **目的**: 達成したい運用業務
- **前提条件**:
  - 対象データの状態
  - 必要な権限ロール
  - 機能フラグ状態
- **トリガー**: 業務指示 / アラート / 定期業務
- **主シナリオ**:
  1. ...画面で対象を検索
  2. ...操作を選択
  3. 理由を入力（監査ログ用）
  4. ドライラン結果を確認
  5. 実行
- **影響範囲**:
  - 何件のレコード
  - どのユーザー
  - どのサブシステムに影響
- **代替シナリオ**: ドライラン / 部分実行 / 巻き戻し
- **異常シナリオ**:
  - 権限不足 → 403、UIに出ない
  - バリデーション失敗
  - 並行操作（同じ対象を別オペレーターが操作中）
- **適用される状況の観点**:
  - `cond.external`: 連携先の障害
  - `cond.time`: バッチ完了時刻
- **観察可能な期待結果**: DB状態 / 監査ログ / 通知
- **検証方法の割り当て**:
  - 自動テスト:
    - integration: DB変更の確認
    - 管理画面 e2e: 主シナリオ
    - 権限境界テスト: 各ロールでの可否
    - 監査ログ検証: 必要項目の記録
  - Agentic Verification (ステージング限定):
    - ブラウザMCPで管理画面操作してドライラン結果確認
    - DB MCP で `audit_logs` を直接 SELECT して記録確認
    - 各ロールでログインして UI/API の実応答確認
- **非対象**: エンドユーザー側の見え方は UC-E-XXX で扱う
```

### 系統O のチェックポイント

- 「影響範囲」が明示されているか（「全ユーザー」のような曖昧な記述になっていないか）
- 監査ログに記録されるべき項目が定義されているか
- 権限境界テストで「権限のないロール」での拒否確認が含まれているか

## 共通の落とし穴

| 落とし穴 | 対処 |
|---|---|
| テンプレートの欄を埋めるのが目的化する | テンプレートはチェックリストです。書くべきものがないなら「該当なし」と明記してください。空欄のまま提出しないでください。 |
| 「Out of scope」を書かない | 何を扱わないかを明記しないと、テストケース展開時に膨れ上がります。 |
| 状況の観点を全UCに重複コピペする | `cond.external` のHTTP 5xx観点は、機能ごとに毎回フルセットで書く必要はありません。共通仕様としてまとめ、UCからは「適用される」と参照してください。 |
