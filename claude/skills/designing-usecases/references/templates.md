# ユースケース記述テンプレート（系統別）

各系統で記述項目が異なります。1つのユースケースは1つの系統に固定します。

引き渡し先の `designing-testcases` が以下のフィールドを使ってテスト計画を立てるため、とくに `系統 / 適用される状況軸 / 観察可能な期待結果 / テスト範囲外` は必ず記入します。

## 系統E テンプレート（エンドユーザー × UI）

```markdown
### UC-E-XXX: <ユースケース名>

- **系統**: E (エンドユーザー × UI)
- **主体**: 例) ログイン済みのエンドユーザー
- **目的**: ユーザーが達成したいこと（プロダクト目的の言葉で書く。技術用語を避ける）
- **前提条件**:
  - 認証状態
  - データ初期状態
  - デバイス・ロケール
- **トリガー**: ユーザー操作 / 通知の受信 / 別画面からの遷移
- **主シナリオ (UI操作)**:
  1. ...画面で...をタップする
  2. ...が表示される
  3. ...
- **代替シナリオ**: 〜のとき: ...
- **異常シナリオ**: 〜が起きたとき、UI上どう見えるか
- **適用される状況軸**:
  - `cond.external`: HTTPステータス全種、遅延
  - `cond.network`: オフライン遷移
- **観察可能な期待結果**: UI上の表示・遷移・トースト等
- **テスト範囲外**: 〜は本UCでは扱わない
```

### 系統E のチェックポイント

- 観察可能な期待結果が「UIから見えるもの」だけになっているか（DB状態を期待結果に書いていないか）
- 主シナリオがユーザーの操作言葉になっているか（API呼び出しの記述になっていないか）
- a11yを「適用される状況軸」ではなく独立した観察対象として扱っているか

## 系統D テンプレート（開発者 × コードAPI）

```markdown
### UC-D-XXX: <APIユースケース名>

- **系統**: D (開発者 × コードAPI)
- **主体**: 例) このコンポーネント/関数を呼び出す開発者
- **目的**: 開発者が実現したい統合（「フォーム送信時に画像をアップロードしたい」等）
- **前提条件**: 依存バージョン / Peer Deps / ランタイム
- **公開シグネチャ**:
  ```ts
  function uploadAvatar(
    file: File,
    options?: UploadOptions
  ): Promise<AvatarResult>
  ```
- 主シナリオ（API呼び出し）:
  1. 必須引数のみで呼ぶ
  2. resolveした値の型と内容
- 代替シナリオ:
  - optionsを渡す
  - 既存値を上書き
  - Suspense境界内で使う
- 異常シナリオ:
  - 不正引数 → 型エラー（コンパイル時）
  - ネットワーク失敗 → rejectの型
  - 中断（AbortSignal）→ rejectの型
- 適用される状況軸:
  - `cond.external`: レスポンス形状の揺れ
  - `cond.network`: AbortSignalの伝搬
- 観察可能な期待結果: 戻り値、型、発火するイベント、副作用、ログ
- テスト範囲外: UI側の表示は別UC (UC-E-XXX) で扱う
```

### 系統D のチェックポイント

- 公開シグネチャが TypeScript で記述されているか
- 型レベルで防げる異常シナリオを「観察可能な期待結果」に入れていないか（実装時に型で担保するため、UCには書かない）
- UI 表示への期待が混入していないか

## 系統O テンプレート（運用者 × 管理画面/CLI）

```markdown
### UC-O-XXX: <運用業務名>

- **系統**: O (運用者 × 管理画面/CLI)
- **主体**: 運用者の役割（CS / SRE / 管理者）と権限ロール
- **目的**: 達成したい運用業務
- **前提条件**:
  - 対象データの状態
  - 必要な権限ロール
  - 機能フラグ状態
- **トリガー**: 業務指示 / アラート / 定期業務
- **主シナリオ**:
  1. ...画面で対象を検索
  2. ...操作を選択
  3. 理由を入力（監査ログ用）
  4. ドライラン結果を確認
  5. 実行
- **影響範囲**:
  - 何件のレコード
  - どのユーザー
  - どのサブシステムに影響
- **代替シナリオ**: ドライラン / 部分実行 / 巻き戻し
- **異常シナリオ**:
  - 権限不足 → 403、UIに出ない
  - バリデーション失敗
  - 並行操作（同じ対象を別オペレーターが操作中）
- **適用される状況軸**:
  - `cond.external`: 連携先の障害
  - `cond.time`: バッチ完了時刻
- **観察可能な期待結果**: DB状態 / 監査ログ / 通知
- **テスト範囲外**: エンドユーザー側の見え方は UC-E-XXX で扱う
```

### 系統O のチェックポイント

- 「影響範囲」が明示されているか（「全ユーザー」のような曖昧な記述になっていないか）
- 監査ログに記録されるべき項目が定義されているか
- 権限境界の観点で「権限のないロール」での拒否確認が含まれているか

## 共通の落とし穴

| 落とし穴 | 対処 |
|---|---|
| テンプレートの欄を埋めるのが目的化する | テンプレートはチェックリストです。書くべきものがないなら「該当なし」と明記します。空欄のまま `designing-testcases` に渡しません。 |
| 「テスト範囲外」を書かない | 何を扱わないかを明記しないと、テスト計画展開時に膨れ上がります。 |
| 状況軸を全UCに重複コピペする | `cond.external` のHTTP 5xx観点は、機能ごとに毎回フルセットで書く必要はありません。共通仕様として `conditions.md` にまとめ、UCからは「適用される」と参照します。 |
