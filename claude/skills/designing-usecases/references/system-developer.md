# 系統D: 開発者（消費者） × コードAPI

## 定義

| 項目 | 内容 |
|---|---|
| 主体 | このシステム/モジュール/コンポーネント/関数を「呼び出す側」の開発者。UIは使わない。 |
| インターフェース | 関数シグネチャ、props、型、戻り値、発火するイベント、副作用 |
| 達成したいこと | 自分のシステムに正しく統合し、リファクタや依存更新で壊れない状態を保つ |
| 観察可能性 | 型エラー、戻り値、副作用、ログ、ビルドサイズ |
| 粒度 | 公開API単位（「`<DatePicker>` を controlled / uncontrolled の両方で使う」「`uploadAvatar()` の戻り値の Promise 型」等） |
| 典型的な失敗モード | 任意プロパティの省略時挙動が不明 / 不正利用が暗黙に動いてしまう / バージョン更新で型が壊れる / Tree Shaking で落ちるべきものが落ちない |
| 主なテスト種別 | unit / 型テスト（tsd, expect-type） / build smoke test / contract test / bundle size assertion |

注: ここでいう「開発者」はこのシステムを消費する側の開発者を指す。「自分自身（実装者）の開発者」観点はユースケースに書かない（実装時の自然な配慮なので、UCにすると過剰生成になる）。

## 観点チェックリスト

### 公開APIの契約

- [ ] 公開シグネチャの全パターンが呼び出せる
- [ ] 必須引数/必須propsを欠いた場合、型エラーが出る
- [ ] オプショナル引数/任意propsの省略時のデフォルト値が定義通り
- [ ] 戻り値の型が宣言通り
- [ ] 引数の型が広すぎない（例: `any` を排除）

### 不正利用への防御

- [ ] 不正な引数を渡した時、ランタイムで明確に失敗する（暗黙に動作するのは危険）
- [ ] 例外として投げる場合、その例外の型・タイミングがドキュメント化されている
- [ ] エラーバウンダリで catch できる粒度になっている
- [ ] silent failure になっていない（例: catch して何もしない関数）

### 副作用と状態遷移

- [ ] 呼び出し順序の制約（A の前に B を呼んだら？）
- [ ] 冪等性（同じ呼び出しを2回したら？）
- [ ] 再呼び出し時の挙動（cleanup されるか）
- [ ] イベント発火の重複・抜け
- [ ] グローバル状態の汚染（テスト間で漏れないか）

### 拡張性

- [ ] children / slot / renderProps が期待通り渡る
- [ ] ref forwarding が動作する
- [ ] カスタムイベントハンドラが上書きでなく合成される
- [ ] スタイル拡張（className / style 上書き / CSS variables）が可能

### 互換性

- [ ] 依存パッケージのバージョン差（peer dependency の最小〜最大）
- [ ] ESM / CJS の両方からimportできる
- [ ] バンドラー間（webpack / Vite / Rspack / esbuild）で動作
- [ ] React 18 / 19 のような major 差
- [ ] Node.js のバージョン互換（CLIなら）

### Tree Shaking と副作用宣言

- [ ] `package.json` の `sideEffects` が正しく宣言されている
- [ ] 不要なコードがバンドルから落ちる（実測）
- [ ] barrel file 経由で全体が引き込まれていない
- [ ] `/*#__PURE__*/` アノテーションが必要箇所に付いている

### 型レベルの保証

- [ ] discriminated union が正しく narrow される
- [ ] generics の制約が機能している
- [ ] `as const` を使う側が困らないか
- [ ] 型レベルで防げるエラーをランタイムテストにしていない（フィードフォワードに寄せる）

### ドキュメントとの整合

- [ ] README / TSDoc の記述と実際の挙動が一致
- [ ] サンプルコードがそのままビルド・実行できる
- [ ] migration guide で挙動差が明記されている（破壊的変更時）

## 検証方法への割り当て指針

各観点について、自動テストとAgentic Verificationの両方の選択肢を持ちます。
詳細な使い分けは `references/verification-methods.md` を参照してください。

| 観点 | 自動テスト | Agentic Verification |
|---|---|---|
| 公開APIの契約（ランタイム） | unit | 最小サンプル書いて実行して挙動確認 |
| 公開APIの契約（型） | tsd / expect-type | `tsc --noEmit` で実エラーを直接確認 |
| 不正利用への防御 | unit（reject パターン） | 不正引数で実呼び出ししてエラー型確認 |
| 副作用と状態遷移 | unit | サンドボックスで実呼び出し、グローバル状態を直接観測 |
| 拡張性 | unit + component test | 実プロジェクトに組み込んで挙動確認 |
| 互換性（ESM/CJS、バンドラー） | build smoke test | 実プロジェクト（webpack / Vite / Rspack）で import して動作確認 |
| Tree Shaking | bundle size assertion | 実バンドル成果物を grep / AST で検査、`@rollup/plugin-visualizer` の出力を読む |
| 型レベルの保証 | 型テストのみ（runtime test にしない） | `tsc` で実コンパイル |
| ドキュメントとの整合 | サンプルコードの自動ビルド | サンプルコードを実プロジェクトに貼って動作確認 |
| SSR `window` ガード | unit (jsdom + node 両環境) | Next.js / Remix 実プロジェクトで SSR 実行、コンソールエラー確認 |

## 系統Dで起きがちな間違い

| 間違い | 対処 |
|---|---|
| 型で表現できることをランタイムテストで書く | テスト過剰生成の典型です。型テストで足りるなら型テストに寄せてください。 |
| 「内部実装の開発者」観点を混ぜる | モジュール内部の実装詳細（プライベート関数の挙動）はユースケース化しません。公開APIの観点に絞ってください。 |
| UI挙動を系統Dで検証する | 描画結果は系統Eの責務です。系統Dは「propsを与えたら、コールバックがこの引数で呼ばれる」までを担います。 |
| 依存バージョンマトリクスを無限に広げる | 公式サポート範囲を明示し、その範囲のみテストしてください。 |
| Tree Shakingを「やってるはず」で済ませる | 実測（bundle analyzer / size assertion）が必要です。 |
