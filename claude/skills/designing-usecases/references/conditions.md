# 横断的な状況・制約

外部システム / 時間 / ランタイム環境 / ネットワークは、ユースケースの主体ではなく、各系統UCに横断的に作用する状況・制約として記述します。
全ての系統（E / D / O）のUCに対して、必要な軸を当ててください。

## `cond.external` — 外部システム

API、Webhook、決済、認証、メッセージング、CDN、ストレージ等の状態を指します。

### 観点チェックリスト

- [ ] レスポンス形状の揺れ（空配列 / null / undefined / 想定外フィールド / フィールド欠損 / 想定外の型）
- [ ] レスポンスの遅延（通常 / 遅い / タイムアウト / 途中切断）
- [ ] HTTPステータス全種（2xx / 3xx / 4xx: 401, 403, 404, 409, 422, 429 / 5xx）
- [ ] ネットワーク層のエラー（DNS失敗、TLSエラー、ECONNRESET）
- [ ] レート制限（429時のリトライ・バックオフ、上限到達時のUX）
- [ ] 認証期限切れ（トークン期限切れ・リフレッシュ失敗・同時リフレッシュ）
- [ ] 冪等性（同じリクエストを2回送ったら2件作成されないか）
- [ ] 順序保証（並行リクエストのレース、後勝ち / 先勝ち）
- [ ] Webhook / push 受信（重複配信、順序逆転、署名検証失敗、ペイロード不整合）
- [ ] 契約テスト（OpenAPI / GraphQL Schema / Pact 等でスキーマ変更を検出）

### 系統別の現れ方

| 系統 | 観察される現象 | 検証方法 |
|---|---|---|
| E | スピナー / タイムアウトメッセージ / 楽観的更新の巻き戻し | e2e でネットワーク絞り |
| D | Promise が pending / AbortSignal の伝搬 / リトライ回数 | unit + fake fetch (MSW等) |
| O | バッチが時間内に完了しない / アラート発火 | integration + 時刻モック |

## `cond.time` — 時間

明示的に意識しないと、テスト時の固定時刻でしか動作確認されません。

### 観点チェックリスト

- [ ] タイムゾーン（ユーザーTZ / サーバーTZ / DB保存TZ の差、UTC を境界としてどう持つか）
- [ ] 夏時間 (DST) の切替日をまたぐ集計、繰り返し予定
- [ ] 境界時刻（日付の0:00、月末、年末、うるう年、うるう秒）
- [ ] 期限（トークン有効期限、キャッシュ TTL、セッションタイムアウト、機能の公開/終了日時）
- [ ] 長時間放置（タブを開いたまま数時間〜数日放置後の操作）
- [ ] クライアント時計のずれ（信用できない前提、サーバー時刻の取得・補正）
- [ ] 同時実行・スケジューリング（cron 起動の重複、スケジュール時刻の前後）

### 系統別の現れ方

| 系統 | 観察される現象 | 検証方法 |
|---|---|---|
| E | 期限切れ通知、再ログイン誘導、日付表示のローカライズ | e2e（時刻モック） |
| D | TTL切れ時のキャッシュ再取得、Token refresh の発火 | unit（fake timers） |
| O | バッチ完了時刻、cron 重複発火、保持期間切れの自動削除 | integration |

## `cond.runtime` — ランタイム環境

ブラウザやJSランタイムが「勝手に振る舞う」もの全般を対象とします。

### 観点チェックリスト

- [ ] ライフサイクルイベント（`visibilitychange`、`pagehide`、`beforeunload`、`focus/blur`、`online/offline`、bfcache 復帰）
- [ ] レンダリング指標（CLS、LCP、INP、フォントスワップ、画像遅延読み込み）
- [ ] SSR / CSR / SSG の境界（ハイドレーション不一致、サーバー専用APIをクライアントで呼ぶ事故、`window` / `document` 参照のガード）
- [ ] ストレージ制約（localStorage / sessionStorage / IndexedDB / Cookie の容量上限、disable 状態、iOS の ITP）
- [ ] デバイス能力差（タッチ / hover / pointer 種別、`prefers-reduced-motion`、`prefers-color-scheme`、低スペック端末）
- [ ] ブラウザ互換（対応ブラウザでの polyfill、CSS feature の Baseline 状況）
- [ ] 権限 API（通知、位置情報、カメラ、クリップボード — 拒否 / 未応答 / 後から取り消し）
- [ ] 印刷・拡大（印刷プレビュー、ページズーム、文字サイズ変更）

### 系統別の現れ方

| 系統 | 観察される現象 | 検証方法 |
|---|---|---|
| E | リロード後の入力復元、bfcache復帰時の表示、CLS による誤タップ | e2e、Lighthouse CI、visual |
| D | SSR時の `window` ガード忘れ、ハイドレーション不一致 | unit (jsdom + node環境両方)、build smoke |
| O | 管理画面のセッション切れ、長時間放置後のタブ復帰 | e2e |

## `cond.network` — ネットワーク

アプリは「常に繋がっている」前提で書きがちです。

### 観点チェックリスト

- [ ] オフライン / オンライン遷移（通信断中に書いたデータ、復帰時の同期）
- [ ] 低速回線（Slow 3G / 2G、スピナー表示、楽観的更新、タイムアウト）
- [ ] リクエスト中断（ナビゲーション中の in-flight、AbortController、router の遷移キャンセル）
- [ ] 再送・重複（同一リクエストの並行発火、連打、useEffect 二重実行）
- [ ] HTTP/2、HTTP/3 特有の挙動（多重化、優先度）
- [ ] CDN / キャッシュ（stale-while-revalidate、キャッシュ無効化、preflight）

### 系統別の現れ方

| 系統 | 観察される現象 | 検証方法 |
|---|---|---|
| E | オフライン時の保存、復帰時の自動送信、低速時のローディング表示 | e2e（network throttle） |
| D | AbortSignal の伝搬、Promise の reject 種別、リトライポリシー | unit（fake fetch） |
| O | 大量更新時の中断・再開、バッチの分割実行 | integration |

## 系統 × 状況のクロス表（早見表）

|  | `cond.external` | `cond.time` | `cond.runtime` | `cond.network` |
|---|---|---|---|---|
| 系統E | エラー時のUI、リトライ動線 | 期限切れトークン時の再ログイン誘導 | リロード後の入力復元 | オフライン時の保存・復帰時の送信 |
| 系統D | レスポンス揺れの型ガード | TTL切れ時のキャッシュ再取得 | SSR時の `window` ガード | AbortSignal の伝搬 |
| 系統O | 連携先障害時のリトライ・通知 | バッチ完了時刻、cron重複 | 管理画面のセッション切れ | 大量更新時の中断・再開 |

## 状況観点を扱う原則

- 全UCに状況観点を網羅して書かないでください。同じ状況パターン（例: HTTP 5xx時の挙動）は共通仕様としてまとめ、各UCからは「適用される」と参照するだけで十分です。
- 状況検証は下位のテスト種別に集約してください。5xx時の挙動はunit / integrationで網羅し、e2eでは1パターンだけ通します。e2eで全状況を試すとテスト時間が爆発します。
- 状況のモック方針は系統ごとに揃えてください。同じシステム内で系統ごとにモックライブラリがバラバラだと保守できません。チームで方針を統一します。
