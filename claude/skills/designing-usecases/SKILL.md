# designing-usecases

「対象者 × インターフェイス」で系統分けし、各系統に観点と適用される状況軸を紐づけたUC表を作ります。出力は`designing-testcases`の入力となります。

## 核となる方針

### 系統分けは「対象者 × インターフェイス」で行う

対象者ごとに触れるインターフェイスが違うため、達成したいこと・観察可能性・失敗の意味・主たるテスト種別がすべて変わります。1つのユースケース表に複数系統を混在させてはいけません。

| 系統 | 対象者 | インターフェイス |
|---|---|---|
| E | エンドユーザー | UI（画面・操作・通知） |
| D | 開発者（消費者） | コードAPI（関数、props、型） |
| O | 運用者・管理者 | 管理画面 / CLI |

各系統の定義・観点チェックリスト・典型的な失敗モードは以下を読んでください。

| 系統 | reference |
|---|---|
| E | `references/system-end-user.md` |
| D | `references/system-developer.md` |
| O | `references/system-operator.md` |

### 非人間アクターはUCの主体にしない

外部システム / 時間 / ランタイム環境 / ネットワークは、各系統UCに横断的に作用する状況・制約として記述します。

| 状況軸ID | 軸 |
|---|---|
| `cond.external` | 外部システム（API、Webhook、決済、認証） |
| `cond.time` | 時間（TZ、DST、期限、TTL、放置） |
| `cond.runtime` | ランタイム環境（ブラウザライフサイクル、SSR/CSR、ストレージ） |
| `cond.network` | ネットワーク（オフライン、低速、中断） |

各状況軸の観点チェックリストと系統別の現れ方は`references/conditions.md`を読んでください。

## タスク判定

| 入力 | タスク |
|---|---|
| 既存のUC表がある | タスクA: UC表のレビュー |
| ゼロから始める | タスクA': UC表の新規作成 |

判定が曖昧な場合は「主たる対象者は誰ですか？（エンドユーザー / 統合する開発者 / 社内オペレーター）」と確認します。

## タスクA: UC表のレビュー

既存のUC表を入力として受け取ります。

### ステップ

1. スコープ判定 — 対象システムのタイプを特定し、読むreferenceを決めます。

   | 対象システムのタイプ | 読むreference |
   |---|---|
   | B2C / B2B プロダクト | `system-end-user.md`, `system-operator.md`, `conditions.md`, `templates.md` |
   | コンポーネントライブラリ / SDK / APIサービス | `system-developer.md`, `conditions.md`, `templates.md` |
   | 管理ツール / 内製運用システム | `system-operator.md`, `conditions.md`, `templates.md` |
   | 全レイヤーを含む大型プロダクト | 全reference |

2. 系統確認 — 各UCの系統が明示されているか確認します。明示なければ内容から推定して指摘します。
3. 混在確認 — 1UCに複数系統が混在していないか確認します。混在していれば分割を提案します。
4. 欠落系統の指摘 — 対象システムのスコープに対して欠けている系統があれば指摘します。
5. 状況軸の確認 — 各UCに`conditions.md`のチェックリストを当て、未カバーの状況軸を列挙します。

### 完了条件

- [ ] ①系統が混在しているUC（分割要）の一覧が出ている
- [ ] ②欠けている系統のUC候補が出ている
- [ ] ③既存UCに追記すべき状況軸が出ている
- [ ] 上記3点がゼロの場合「カバレッジ問題なし」と明記されている
- [ ] 各UCが`references/templates.md`のスキーマを満たしている

完了条件を満たしたら結果をユーザーに提示し、承認を得てから`designing-testcases`に引き渡します。

## タスクA': UC表の新規作成

機能仕様または未整理の要件を入力として受け取り、UC表をゼロから作成します。

### ステップ

1. 対象機能の確認 — 機能名、解決したい課題、想定される利用者を確認します。曖昧なら対話で詰めます。
2. 系統の列挙 — タスクAのステップ1と同じく、対象システムのタイプから関わる系統を列挙します。
3. 系統ごとにUCを起こす — 各系統について`references/templates.md`のテンプレートにしたがってUCを記述します。
4. 状況軸の付与 — 各UCに`references/conditions.md`から該当する状況軸を当てます。
5. 重複・矛盾確認 — 系統間で重複した観察対象や矛盾する期待結果がないか確認します。

### 完了条件

- [ ] 関わる全系統についてUCが作成されている
- [ ] 各UCが`references/templates.md`のスキーマを満たしている
- [ ] 状況軸が各UCに紐づいている
- [ ] 「テスト範囲外」が各UCに明示されている

完了条件を満たしたらUC表をユーザーに提示し、承認を得てから`designing-testcases`に引き渡します。

## 引き渡しスキーマ

各UCが`references/templates.md`のスキーマを満たした状態で`designing-testcases`に引き渡します。

## 共通の落とし穴

| 落とし穴 | 対処 |
|---|---|
| 「対象者: エンドユーザー / 開発者」だけで分類する | 状況軸（`cond.xxx`）の記述場所がなくなります。状況軸を必ず併用してください。 |
| 1UCに複数系統を混ぜる | UI操作のステップとAPI呼び出しの戻り値が同じUCに書かれていたら必ず分割します。 |
| 状況軸を全UCに重複コピペする | `cond.external`のHTTP 5xx観点は、共通仕様として`conditions.md`を参照する形で十分です。 |
| 系統Dで型で済むものを観察対象にする | 型レベルで防げるものはUCの観察対象に書きません（実装時に自然に型で担保されます）。 |
| 系統Oを忘れる | B2Cプロダクトでも運用業務は必ず存在します。スコープに入るかを最初に確認してください。 |
| 「観察可能な期待結果」に内部状態を書く | UCの観察対象は系統に対応するインターフェイスから見えるものに限定します。 |
