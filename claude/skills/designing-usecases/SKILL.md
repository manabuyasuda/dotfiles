# designing-usecases

テストの抜け漏れを防ぐためのユースケース設計スキルです。主体とインターフェースで系統を分け、各系統に対して検証3層（自動テスト / Agentic Verification / 手動）を割り当てます。

責務は「ユースケースを設計し、テスト計画を作ること」です。テストコードの実装はタスクCの手順に従い実施します。

## 核となる方針（必読）

### ユースケースは「対象者 × インターフェース」で系統分けする

対象者ごとに触れるインターフェースが違うため、達成したいこと・観察可能性・失敗の意味・主たるテスト種別がすべて変わります。1つのユースケース表に複数系統を混在させてはいけません。

| 系統 | 主体 | インターフェース | 主なテスト種別 |
|---|---|---|---|
| E | エンドユーザー | UI（画面・操作・通知） | e2e / visual / a11y |
| D | 開発者（消費者） | コードAPI（関数、props、型） | unit / 型テスト / contract / build smoke |
| O | 運用者・管理者 | 管理画面 / CLI | integration / 管理画面e2e / 権限境界テスト |

### 非人間アクターはユースケースの主体にしない

外部システム / 時間 / ランタイム環境 / ネットワークは、各系統UCに横断的に作用する状況・制約として記述します。

| 状況ID | 軸 |
|---|---|
| `cond.external` | 外部システム（API、Webhook、決済、認証） |
| `cond.time` | 時間（TZ、DST、期限、TTL、放置） |
| `cond.runtime` | ランタイム環境（ブラウザライフサイクル、SSR/CSR、ストレージ） |
| `cond.network` | ネットワーク（オフライン、低速、中断） |

### 検証は3層で考える

| 層 | 主な目的 | 例 |
|---|---|---|
| 自動テスト | 退行検知（CIで繰り返す） | unit / e2e / visual / contract |
| Agentic Verification | 仕様確定 / 不具合再現 / 実環境の実値確認 | ブラウザMCP操作、`getComputedStyle()` で実値取得 |
| 手動検証 | UX判断 / 最終受け入れ | 実機でのスクリーンリーダー確認 |

3層は補完関係です。Agentic Verification で確認したことを必要に応じて自動テストに昇格させます。

## タスク判定

ユーザーが求めているものを最初に判定し、対応するタスクのプロセスに進む。

| ユーザーの入力 | タスク |
|---|---|
| 既存のユースケース表がある | タスクA: ユースケース表のレビュー |
| 系統が確定したUCがある | タスクB: テスト計画の生成 |
| テスト計画があり実装に進む | タスクC: テストコードの実装 |
| 何もない（ゼロから始める） | タスクA → タスクB → タスクC の順に進む |

判定が曖昧な場合は「主たる対象者は誰ですか？（エンドユーザー / 統合する開発者 / 社内オペレーター）」と確認してください。

## タスクA: ユースケース表のレビュー

既存のユースケース表を入力として受け取ります。

### ステップ

1. スコープ判定 — 対象システムのタイプを特定し、読むreferenceを決めます。

   | 対象システムのタイプ | 読むreference |
   |---|---|
   | B2C / B2B プロダクト | `system-end-user.md`, `system-operator.md`, `conditions.md`, `templates.md`, `verification-methods.md` |
   | コンポーネントライブラリ / SDK / APIサービス | `system-developer.md`, `conditions.md`, `templates.md`, `verification-methods.md` |
   | 管理ツール / 内製運用システム | `system-operator.md`, `conditions.md`, `templates.md`, `verification-methods.md` |
   | 全レイヤーを含む大型プロダクト | 全reference |

2. 系統確認 — 各UCの系統が明示されているか確認します。明示なければ内容から推定して指摘します。
3. 混在確認 — 1UCに複数系統が混在していないか確認します。混在していれば分割を提案します。
4. 欠落系統の指摘 — 対象システムのスコープに対して欠けている系統があれば指摘します。
5. 状況観点の確認 — 各UCに `conditions.md` のチェックリストを当て、未カバーの状況観点を列挙します。

### 完了条件

- [ ] ①系統が混在しているUC（分割要）の一覧が出ている
- [ ] ②欠けている系統のUC候補が出ている
- [ ] ③既存UCに追記すべき状況観点が出ている
- [ ] 上記3点がゼロの場合「カバレッジ問題なし」と明記されている

## タスクB: ユースケースからテスト計画を生成

タスクAの出力、またはユーザーが持ち込んだ、系統が確定したUCを入力として受け取ります。

### ステップ

1. 主たるテスト種別の決定 — UCの系統に応じて §核となる方針 §1.1 から決めます。
2. 状況観点の抽出 — UCの「適用される状況」ごとに `conditions.md` から具体的な検証項目を抽出します。
3. 検証手段の割り当て — 各検証項目に対して `verification-methods.md` を参照し、3層のどれで担保するかを割り当てます。1観点に複数層を割り当ててかまいません。
4. テスト計画の展開 — 下記テスト計画テンプレートに従って出力します。
5. 重複確認 — 同じ機能の異なる系統UCが両方ある場合、重複に見える項目でも省略しません（観察対象が違うため両方必要です）。
6. 集約 — 状況観点の検証は下位のテスト種別に1回集約します（例: 5xx は unit で網羅し、e2eは1パターンのみ）。

### テスト計画テンプレート

```markdown
## テスト計画: [機能名]

### 系統 [E/D/O]: [系統名]

| 検証項目 | 状況観点 | 自動テスト | Agentic Verification | 手動 |
|---|---|---|---|---|
| [項目] | [cond.xxx] | [テスト種別: 内容] | [手段: 内容] | [手順] |

**退行検知が必要でAgentic Verificationのままでよい項目**:
- [理由と根拠]

**自動テストへの昇格を推奨する項目**:
- [昇格基準に該当する理由]
```

### 完了条件

- [ ] 各系統について主たるテスト種別が明記されている
- [ ] 状況観点（cond.xxx）が各UCに紐づいている
- [ ] 各検証項目に3層のいずれかが割り当てられている
- [ ] 「自動テストへの昇格推奨」と「Agenticのまま残す」の判断が明記されている

## タスクC: テストコードの実装

タスクBで生成したテスト計画を入力として受け取ります。

### 前提となる設計判断

- フィードフォワード（型/Lint/契約）とフィードバック（runtime test）の2軸を意識してください。系統Dでは特に、型で表現できることはテストにせず型に寄せます。
- 同じ機能の異なる系統のテストは別ファイルに分けます（混在させると失敗時の原因系統が不明瞭になります）。
- 状況観点のモックは、テスト種別に応じて選びます。
  - unit → モック関数
  - integration → contract / 実モック（MSW等）
  - e2e → 実物 or stable testenv（必要なら network throttle）

### 避けるべきパターン

- 1つのe2eテストで複数の状況観点を同時に検証する（失敗時の切り分けが困難）
- 系統Eのテストで実装詳細（内部state、DOM構造の細部）にアサーションする
- 系統Dのテストでブラウザ上のUI表示を確認する（それは系統Eの責務）

### 完了条件

- [ ] テストが系統ごとに別ファイルに分かれている
- [ ] 型で防げる異常シナリオがruntimeテストに紛れ込んでいない（系統D）
- [ ] 各テストが単一の検証項目に対応している（e2eで複数状況を同時検証していない）

## 共通の落とし穴

| 落とし穴 | 対処 |
|---|---|
| 「対象者: エンドユーザー / 開発者」だけで分類する | 状況軸（`cond.xxx`）の記述場所がなくなります。§1.2 の状況軸を必ず併用してください。 |
| 1UCに複数系統を混ぜる | UI操作のステップとAPI呼び出しの戻り値が同じUCに書かれていたら必ず分割してください。 |
| 状況観点を全UCに重複コピペする | `cond.external` の HTTP 5xx 観点は、共通仕様として `conditions.md` を参照する形で十分です。 |
| 系統Dで型で済むものをテストにする | フィードフォワードに寄せてください。 |
| 系統Oを忘れる | B2Cプロダクトでも運用業務は必ず存在します。スコープに入るかを最初に確認してください。 |
| Agentic Verificationを退行検知に使う | 仕様確定・探索が本領です。退行検知が必要なものは自動テストへの昇格を検討してください。 |
