# 系統D: 開発者（消費者） × コードAPI

## 定義

| 項目 | 内容 |
|---|---|
| 主体 | このシステム/モジュール/コンポーネント/関数を「呼び出す側」の開発者。UIは使わない。 |
| インターフェイス | 関数シグネチャ、props、型、戻り値、発火するイベント、副作用 |
| 達成したいこと | 自分のシステムに正しく統合し、リファクターや依存更新で壊れない状態を保つ |
| 観察可能性 | 型エラー、戻り値、副作用、ログ、ビルドサイズ |
| 粒度 | 公開API単位（「`<DatePicker>` を controlled / uncontrolled の両方で使う」「`uploadAvatar()` の戻り値の Promise 型」等） |
| 典型的な失敗モード | 任意プロパティの省略時挙動が不明 / 不正利用が暗黙に動いてしまう / バージョン更新で型が壊れる / Tree Shaking で落ちるべきものが落ちない |

注: ここでいう「開発者」はこのシステムを消費する側の開発者を指します。「自分自身（実装者）の開発者」観点はユースケースに書きません（実装時の自然な配慮なので、UCにすると過剰生成になります）。

## 観点チェックリスト

### 公開APIの契約

- [ ] 公開シグネチャの全パターンが呼び出せる
- [ ] 必須引数/必須propsを欠いた場合、型エラーが出る
- [ ] オプショナル引数/任意propsの省略時のデフォルト値が定義通り
- [ ] 戻り値の型が宣言通り
- [ ] 引数の型が広すぎない（例: `any` を排除）

### 不正利用への防御

- [ ] 不正な引数を渡した時、ランタイムで明確に失敗する（暗黙に動作するのは危険）
- [ ] 例外として投げる場合、その例外の型・タイミングがドキュメント化されている
- [ ] エラーバウンダリでcatchできる粒度になっている
- [ ] silent failureになっていない（例: catchして何もしない関数）

### 副作用と状態遷移

- [ ] 呼び出し順序の制約（Aの前にBを呼んだら？）
- [ ] 冪等性（同じ呼び出しを2回したら？）
- [ ] 再呼び出し時の挙動（cleanupされるか）
- [ ] イベント発火の重複・抜け
- [ ] グローバル状態の汚染（テスト間で漏れないか）

### 拡張性

- [ ] children / slot / renderPropsが期待通り渡る
- [ ] ref forwardingが動作する
- [ ] カスタムイベントハンドラーが上書きでなく合成される
- [ ] スタイル拡張（className / style上書き / CSS variables）が可能

### 互換性

- [ ] 依存パッケージのバージョン差（peer dependencyの最小〜最大）
- [ ] ESM / CJSの両方からimportできる
- [ ] バンドラー間（webpack / Vite / Rspack / esbuild）で動作
- [ ] React 18 / 19のようなmajor差
- [ ] Node.jsのバージョン互換（CLIなら）

### Tree Shaking と副作用宣言

- [ ] `package.json` の `sideEffects` が正しく宣言されている
- [ ] 不要なコードがバンドルから落ちる（実測）
- [ ] barrel file経由で全体が引き込まれていない
- [ ] `/*#__PURE__*/` アノテーションが必要箇所に付いている

### 型レベルの保証

- [ ] discriminated unionが正しくnarrowされる
- [ ] genericsの制約が機能している
- [ ] `as const` を使う側が困らないか
- [ ] 型レベルで防げるエラーはUCの観察対象に書かない（実装時に型で担保する）

### ドキュメントとの整合

- [ ] README / TSDocの記述と実際の挙動が一致
- [ ] サンプルコードがそのままビルド・実行できる
- [ ] migration guideで挙動差が明記されている（破壊的変更時）

## 系統Dで起きがちな間違い

| 間違い | 対処 |
|---|---|
| 「内部実装の開発者」観点を混ぜる | モジュール内部の実装詳細（プライベート関数の挙動）はUC化しません。公開APIの観点に絞ります。 |
| UI挙動を系統Dで検証する | 描画結果は系統Eの責務です。系統Dは「propsを与えたら、コールバックがこの引数で呼ばれる」までを担います。 |
| 依存バージョンマトリクスを無限に広げる | 公式サポート範囲を明示し、その範囲のみ観点に入れます。 |
| Tree Shakingを「やってるはず」で済ませる | 実測（bundle analyzer / size assertion）の方針を観察可能な期待結果に入れます。 |
| 型で防げるものをランタイムの観点として書く | 型レベルで防げるなら観察対象にしません。実装時に型に寄せる方針を `designing-testcases` に伝えます。 |
