# 系統E: エンドユーザー × UI

## 定義

| 項目 | 内容 |
|---|---|
| 主体 | プロダクトの利用者。コードの中身は知らない。誤操作も中断もする。 |
| インターフェイス | 画面、操作、通知 |
| 達成したいこと | ユーザー自身の目的（情報取得、タスク完遂、購入等） |
| 観察可能性 | UI上の表示・状態遷移・フィードバック。本人は「裏で何が起きたか」を見られない。 |
| 粒度 | ジャーニー単位（「サインアップして最初の投稿をする」）または機能単位（「プロフィール画像を変える」） |
| 典型的な失敗モード | 何をすればいいかわからない / 完了したか確信が持てない / 失敗時に次の行動が分からない |

## 観点チェックリスト

各UCに対し、該当する項目をすべて確認します。

### 主要ジャーニー

- [ ] 正常系を最後まで通せる
- [ ] 完了したかどうかがUI上で明確（成功フィードバック、状態の更新が見える）
- [ ] 途中状態がわかる（ローディング、進捗、保留中バッジ）

### 入力のバリエーション

- [ ] 境界値: 空 / 1文字 / 最大長 / 最大長+1 / 0 / 負数 / 最大値 / 最大値+1
- [ ] 異常値: 制御文字、絵文字、サロゲートペア、RTL文字、ゼロ幅文字
- [ ] ロケール: 数値フォーマット（`1,234.56` vs `1.234,56`）、日付フォーマット、漢字/かな/全角半角、通貨記号
- [ ] 形式違反: 不正なメールアドレス、不正な電話番号、不正なURL
- [ ] ペースト: 改行混入、整形済みリッチテキスト、HTMLタグ混入

### 中断と再開

- [ ] ページリロード後、入力が復元されるべきか / されなくてよいか
- [ ] ブラウザバックからの復帰
- [ ] タブ切替後の戻り
- [ ] 別タブで同じ操作を並行実行
- [ ] アプリのバックグラウンド化（モバイル）

### エラーリカバリ

- [ ] 失敗時、何が起きたかがユーザーの言葉で伝わる（技術用語ではなく）
- [ ] 次の行動が示される（再試行 / 別の手段 / 問い合わせ動線）
- [ ] エラー後、入力が消えていない
- [ ] 部分成功時、どこまで成功したかがわかる

### アクセシビリティ

- [ ] キーボードのみで完遂できる
- [ ] スクリーンリーダーで意味が伝わる（aria-label, ライブリージョン）
- [ ] フォーカス順序が論理的
- [ ] フォーカストラップ（モーダル等）
- [ ] コントラスト比（WCAG AA以上）
- [ ] 色だけに頼らない（エラー赤＋アイコン＋文言）
- [ ] `prefers-reduced-motion` 尊重

### デバイス多様性

- [ ] モバイル / タブレット / デスクトップ
- [ ] タッチ / マウス / キーボード入力
- [ ] 縦・横画面
- [ ] 小さい画面（〜375px）/ 大きい画面（〜1920px超）
- [ ] 文字サイズ拡大（ブラウザ設定で200%）

### 状態の差

- [ ] ログイン前 / ログイン後
- [ ] 無料 / 有料 / 試用中
- [ ] 初回 / リピート（オンボーディング有無）
- [ ] 未認証 / 認証済み（メール認証等）
- [ ] 機能フラグON / OFF
- [ ] 空状態 / 通常 / 大量データ

### 悪意のない誤操作

- [ ] 連打（ダブルサブミット防止）
- [ ] 戻るボタン連打
- [ ] フォーム送信中のリロード
- [ ] 複数タブで同じ更新を並行実行
- [ ] 「保存せずに離脱」時の警告

## 系統Eで起きがちな間違い

| 間違い | 対処 |
|---|---|
| 実装詳細を観察対象にする | 内部state、特定のCSSクラス、特定のDOM構造の細部に依存すると、リファクターで壊れます。観察対象は「ユーザーから見える振る舞い」に限定します。 |
| 1UCで多くを検証しすぎる | 失敗時に原因が分からなくなります。1UC1ジャーニーが原則です。 |
| a11yを後回しにする | 設計段階で観点に入れていないと、後から追加するコストが高くなります。最初のUCから含めます。 |
